02

7 lines. 3 modes. Zero config. 7 行代码。3 种模式。零配置。

Here's the complete code. Not a snippet — the entire caster. 这是完整代码。不是片段 — 是整个 Caster。

Python · my_caster.py
from rune import Caster, RuneContext

caster = Caster("localhost:50070")

@caster.rune("echo", gate="/echo")
async def echo(ctx: RuneContext, input: bytes) -> bytes:
    return input

caster.run()

That @caster.rune("echo", gate="/echo") decorator does everything: registers with the runtime via gRPC, declares the HTTP route, and makes the function available in all three modes. Here's the proof: 这个 @caster.rune("echo", gate="/echo") 装饰器做了所有事：通过 gRPC 注册到 Runtime、声明 HTTP 路由、让函数以三种模式可用。以下是证明：

Sync — immediate response同步 — 立即响应

$ curl -X POST localhost:50060/echo -d '{"message":"hello rune"}' # {"message":"hello rune"}

Stream — SSE events as they're produced流式 — SSE 事件实时产出

$ curl -X POST "localhost:50060/echo?stream=true" -d '{"message":"hello rune"}' # event: stream # data: {"message":"hello rune"} # event: end # data: {}

Async — fire and poll异步 — 提交后轮询

$ curl -X POST "localhost:50060/echo?async=true" -d '{"message":"hello rune"}' # {"task_id":"a1b2c3d4-...","status":"pending"} $ curl localhost:50060/api/v1/tasks/a1b2c3d4-... # {"task_id":"a1b2c3d4-...","status":"completed","result":{"message":"hello rune"}}
Same function. Same config. Same error handling. The runtime handles the protocol mechanics. Your code doesn't even know which mode it's running in. 同一个函数。同一份配置。同一套错误处理。 Runtime 处理协议机制。你的代码甚至不知道自己在以哪种模式运行。
03

What's in the box 盒子里是什么

Rune is a Rust runtime of 10 crates, layered by protocol boundaries. Here's the request path: Rune 是由 10 个 Rust crate 组成的运行时，按协议边界分层。以下是请求路径：

Client
Any HTTP client — curl, browser, your service. Sends a POST request.任意 HTTP 客户端 — curl、浏览器、你的服务。发送 POST 请求。
Gate
Axum-based HTTP layer. Auth, rate limiting, schema validation, file uploads. Routes to the right Rune.基于 Axum 的 HTTP 层。认证、限流、Schema 校验、文件上传。路由到正确的 Rune。
rune-gate · rune-schema
Relay
Global registry. Maps rune names to live invokers. Maintains routing indexes for business routes and label-based filtering.全局注册表。将 Rune 名称映射到活跃的 Invoker。维护业务路由索引和标签过滤。
rune-core::relay
Resolver
Picks the target from multiple Casters registered under the same rune name. Strategies: round-robin, random, least-load, priority, label-match.从同名 Rune 的多个 Caster 实例中选择目标。策略：轮询、随机、最小负载、优先级、标签匹配。
rune-core::resolver
Invoker
Sends the execution request over the bidirectional gRPC stream. Manages timeouts, cancellation, and semaphore permits.通过双向 gRPC 流发送执行请求。管理超时、取消和信号量许可。
rune-core::invoker
Caster
Your code. A Python/TypeScript/Rust process running your handler functions. Connected via Rune Wire Protocol (gRPC bidirectional stream).你的代码。一个运行你处理函数的 Python/TypeScript/Rust 进程。通过 Rune Wire Protocol（gRPC 双向流）连接。
SDK · rune-proto

04

True polyglot.
Not "we have a REST API." 真正的多语言。
不是"我们有 REST API"那种。

"Multi-language support" usually means "you can call our REST API from any language." That's not polyglot — that's HTTP. Rune's polyglot means your Python function, TypeScript function, and Rust function all connect to the same runtime over the same protocol, with the same semantics. And they can be steps in the same workflow. "多语言支持"通常意味着"你可以用任意语言调用我们的 REST API"。那不叫多语言 — 那叫 HTTP。Rune 的多语言是指你的 Python 函数、TypeScript 函数和 Rust 函数全部通过同一协议连接到同一 Runtime，具有相同的语义。它们还可以是同一个工作流中的步骤。

Python
@caster.rune("echo",
  gate="/echo")
async def echo(ctx, input):
    return input
TypeScript
caster.rune({
  name: "ts-echo",
  gate: { path: "/ts/echo" },
}, async (ctx, input) => {
  return input;
});
Rust
caster.rune(
  RuneConfig {
    name: "rs-echo",
    gate: Some("/rs/echo"),
    ..Default::default()
  },
  |_ctx, input| async {
    Ok(input)
  },
)?;

05

Explicit contracts,
not "best effort." 显式契约，
而非"尽力而为"。

Most runtimes have behaviors. Rune has guarantees — 18 of them, documented, tested, and contractual. If any is violated, it's a bug. Not a known issue, not a trade-off. A bug. Here are the 5 that change how you think about function execution: 大多数运行时有行为。Rune 有保证 — 18 条，有文档、有测试、有契约。如果任何一条被违反，那是 bug。不是已知问题，不是权衡，是 bug。以下是改变你对函数执行认知的 5 条：

3
Real streaming真流式
The first SSE event from a ?stream=true request comes from actual handler execution, not from post-completion chunking. If your handler takes 5 seconds to produce its first output, the first event arrives after 5 seconds — not after the full execution completes.?stream=true 请求的第一个 SSE 事件来自 handler 实际执行，不是执行完再分块。如果你的 handler 需要 5 秒产出第一个输出，第一个事件就在 5 秒后到达 — 而非全部执行完之后。
vs. Lambda: no native SSE. vs. Workers: body must complete first.
4
Timeout convergence超时收敛
After a request times out, the runtime state — pending map, semaphore permits — is fully cleaned up. The caller receives a Timeout error. No resource leaks, no phantom permits, no orphaned requests.请求超时后，Runtime 状态 — pending map、信号量许可 — 被完全清理。调用方收到 Timeout 错误。没有资源泄漏，没有幽灵许可，没有孤儿请求。
vs. Lambda: timeout kills container, no state cleanup. vs. Workers: CPU limit just terminates.
8
Strict backpressure严格背压
The max_concurrent declared by a Caster is enforced via semaphore. If all permits are consumed, new requests return 503 Unavailable. Permit count never exceeds the declared maximum, regardless of error paths.Caster 声明的 max_concurrent 通过信号量严格执行。所有许可耗尽时，新请求返回 503 Unavailable。许可数永远不超过声明的最大值，无论走哪条错误路径。
vs. Lambda: throttling is account-wide, not per-function. vs. Workers: no built-in concurrency limit.
11
Route conflicts are hard errors路由冲突是硬错误
Two different rune names declaring the same gate.path + method causes a registration failure. The runtime won't silently pick one. Same rune name from multiple Casters is fine — they share the route via the Resolver strategy.两个不同的 Rune 声明相同的 gate.path + method 会导致注册失败。Runtime 不会静默选一个。同名 Rune 来自多个 Caster 没问题 — 它们通过 Resolver 策略共享路由。
vs. Lambda: API Gateway overwrites silently. vs. Workers: last deploy wins.
18
Graceful drain优雅关停
On shutdown signal: /health returns 503 (signal load balancers), new requests are rejected, in-flight requests complete up to the drain timeout. No request is silently dropped during deployment.收到关停信号：/health 返回 503（通知负载均衡器），拒绝新请求，在途请求在 drain 超时内完成。部署期间不会静默丢弃任何请求。
vs. Lambda: cold restart is a new container. vs. Workers: no drain mode.
All 18 guarantees are documented in docs/protocol-guarantees.md and locked by regression tests. Execution semantics, state convergence, backpressure, routing, flow isolation, schema gate, upload gate, label strictness — each with a clear contract and a test that breaks if the contract is violated. 全部 18 条保证记录在 docs/protocol-guarantees.md，由回归测试锁定。执行语义、状态收敛、背压、路由、Flow 隔离、Schema 门控、上传门控、标签严格性 — 每条都有清晰契约和对应的断言测试。

DAG workflows, built in. 内置 DAG 工作流。

You don't need Step Functions or a separate orchestration tool. Rune's Flow engine executes DAG workflows natively, using the same Invoker interface as direct calls. A Rune behaves identically whether called directly or as a flow step. 你不需要 Step Functions 或单独的编排工具。Rune 的 Flow 引擎原生执行 DAG 工作流，使用与直接调用相同的 Invoker 接口。一个 Rune 无论直接调用还是作为 Flow 步骤，行为完全一致。

# Rune v1.0.0 Design Spec

> 基于 v0.1.0（首个正确、可交付版本），规划真正可用的 v1.0.0。
> 目标：让 Rune 成为个人能力积累和复用平台——把开发工作拆分成小模块（Rune 函数），沉淀到能力池中，跨项目复用。

## Context

v0.1.0 完成了核心正确性：

- Rust Runtime（session 管理、invoker 统一抽象、relay 注册表、gate HTTP 网关）
- Python SDK（pip-installable，asyncio 原生）
- 协议稳定（9 种消息类型，13 条 protocol guarantees）
- sync / real stream / async 三种调用模式
- 最小顺序 Chain Flow

v0.1.0 的 Non-Goals 构成 v1.0.0 的主要工作项：认证、持久化、文件传输、Schema 校验、DAG Flow、TypeScript SDK、CLI、配置系统、可观测性、高级调度。

## Design Principles

1. **自用优先**：所有设计决策面向个人开发者使用场景，不做多租户、不做企业级
2. **单二进制部署**：Runtime + CLI 编译为一个二进制，本地和云上用同一个
3. **渐进增强**：v0.1.0 核心不大改，逐版本叠加能力
4. **协议向后兼容**：新字段 optional，老 SDK 不受影响
5. **SDK 与 Runtime 同版本号**：SDK 和 Runtime 使用相同版本号强绑定，同一版本保证完全兼容

## Deployment Model

- **本地开发**：`rune start --dev`（跳过认证，绑定 127.0.0.1）
- **云服务器**：正常配置文件启动，认证开启，绑定 0.0.0.0
- CLI 通过 HTTP API 与 Runtime 通信，可管理本地或远程 Runtime

## Module 1: Authentication & Security

### Design

两层 API Key 机制：

1. **Caster Key** — Caster 注册时携带
   - Runtime 维护 key 列表（存 SQLite）
   - gRPC Attach 时验证，失败拒绝连接
   - CLI 管理：`rune key create / list / revoke`

2. **Gate Key** — HTTP 调用时携带
   - Header: `Authorization: Bearer <key>`
   - 支持多个 key（不同项目用不同 key，便于追踪来源）
   - 可配置某些路由免认证（如 `/health`）

3. **本地开发模式**
   - `rune start --dev` 跳过所有认证
   - 仅绑定 127.0.0.1

4. **CORS 配置**
   - 配置文件中声明允许的 origin
   - dev 模式默认 allow all

### Key 吊销行为

- Key 吊销后，Runtime 主动断开该 Key 对应的所有活跃 gRPC 连接（发送 DetachRequest）
- Gate Key 吊销后，后续使用该 Key 的 HTTP 请求立即返回 `401`

### Implementation

- `rune-core` 新增 `auth` 模块
- Gate 中间件层拦截，认证失败返回 `401 Unauthorized`
- Key 哈希存储（不存明文）

## Module 2: Persistence

### Design

SQLite 单文件数据库，新增 `rune-store` crate。

**存储内容：**

1. **api_keys 表**
   - id, key_hash, key_type(caster/gate), label, created_at, revoked_at

2. **tasks 表**（替代当前内存 DashMap）
   - task_id, rune_name, status, input, output, error
   - created_at, started_at, completed_at
   - 支持查询：按状态筛选、按时间范围、分页

3. **call_logs 表**
   - request_id, rune_name, mode(sync/stream/async), caster_id
   - latency_ms, status_code, input_size, output_size, timestamp

4. **rune_snapshots 表**
   - 记录哪些 Rune 曾注册过、最近在线时间
   - 快照保存最后一次注册时的完整 RuneDeclaration（schema、gate_path、supports_stream 等）
   - Caster 下线后仍可查看完整历史信息

### Implementation

- 新增 `runtime/rune-store/` crate
- 使用 `rusqlite` crate
- 数据库路径：`~/.rune/rune.db`（可配置）
- Runtime 启动时自动建表/迁移
- 调用日志保留天数可配置（默认 30 天）

## Module 3: File Transfer

### Design

支持 Rune 接收和返回二进制/文件数据。

1. **上传（入方向）**
   - Gate 支持 `multipart/form-data`
   - 文件存到临时目录，Rune handler 拿到文件路径 + 元数据
   - 同时支持 JSON body 传参（文件 + 结构化参数混合）
   - 临时文件在请求完成后自动清理

2. **下载（出方向）**
   - Rune 返回文件时，Gate 自动设置 `Content-Disposition`
   - 支持返回 JSON 或 binary，由 Rune 声明决定

3. **协议层扩展**
   - proto 新增 `ExecuteRequest.attachments` 字段
   - `message FileAttachment { string filename, bytes data, string mime_type }`
   - 文件通过 gRPC 传输，单文件上限 4MB（gRPC 默认限制）
   - 超过 4MB 的文件走文件中转 API：
     - Gate 先存到 Runtime 临时目录，生成 file_id
     - file_id 放入 attachments（替代 bytes data）
     - 远程 Caster 通过 `GET /api/v1/files/:id` 下载
     - 本地 Caster 可直接读文件路径（SDK 自动判断）
     - 文件在请求完成后自动清理

4. **SDK 支持**
   - Python: handler 签名 `async def parse(ctx, input, files: list[File])`
   - TypeScript: handler 签名 `async (ctx, input, files: File[]) => ...`
   - `File` 对象：`name, path, mime_type, size`

### Size Limits

- 配置项 `[gate] max_upload_size_mb = 50`
- 超限在 Gate 层拦截，返回 `413 Payload Too Large`

## Module 4: Schema Validation

### Design

恢复 proto 中的 `input_schema` / `output_schema` 字段，Runtime 真实执行校验。

1. **声明方式**
   - Caster 注册时在 `RuneDeclaration` 中携带 JSON Schema
   - Python SDK 装饰器：
     ```python
     @caster.rune(
       name="pdf-parse",
       input_schema={"type": "object", "properties": {"lang": {"type": "string"}}, "required": ["lang"]},
       output_schema={...}
     )
     ```
   - 不声明则跳过校验（向后兼容）

2. **校验时机**
   - Gate 收到 HTTP 请求时校验 input（路由到 Caster 之前）
   - Caster 返回结果时校验 output（返回给调用方之前）
   - 校验失败返回 `422 Unprocessable Entity` + 详细错误信息

3. **OpenAPI 自动生成**
   - `GET /api/v1/openapi.json`
   - 基于所有已注册 Rune 的 gate_path + schema 自动生成
   - Rune 注册/注销时实时更新
   - OpenAPI 只反映**当前在线**的 Rune（Caster 下线后对应路径从 OpenAPI 中移除）

4. **Schema 在 Flow 中的作用**
   - DAG 编排时，上游 output_schema 与下游 input_schema 做静态兼容性检查
   - 注册 Flow 时即可发现不兼容的步骤连接

### Implementation

- 使用 `jsonschema` crate（Rust JSON Schema 校验）
- Schema 存储在 Relay 的 RuneEntry 中
- OpenAPI 生成使用 `utoipa` 或手动拼装 OpenAPI 3.0 JSON

## Module 5: DAG Flow Engine

### Design

从顺序 Chain 升级为完整 DAG。

#### DAG 定义

每个 Flow 是一个有向无环图。节点 = Rune 调用（step），边 = 数据依赖。支持 JSON/YAML 定义 + API 注册。

```yaml
name: doc-pipeline
steps:
  - name: extract
    rune: pdf-parse
  - name: analyze
    rune: doc-analyze
    depends_on: [extract]
  - name: translate
    rune: translate
    depends_on: [extract]
  - name: merge
    rune: report-gen
    depends_on: [analyze, translate]
```

#### 执行语义

- 无依赖的 step 自动并行（`tokio::JoinSet`）
- 所有上游完成后下游才启动
- 任意 step 失败 → 整个 Flow 失败（未启动的下游不执行）

#### 条件分支

```yaml
- name: heavy-analysis
  rune: deep-analyze
  depends_on: [extract]
  condition: "extract.output.page_count > 10"
```

- 基于上游 output 的字段值判断是否执行
- 条件为 false 的 step 标记为 `skipped`
- **Skip 传播规则**：下游 step 的 `depends_on` 中如果有 skipped step，该依赖视为"已满足"（不阻塞下游）。只有当所有上游都 skipped 时，下游才自动 skip
- **input_mapping 对 skipped step 的处理**：引用 skipped step 的 output 时值为 `null`。如果 input_mapping 中有必填字段引用了 skipped step，注册 Flow 时不报错，执行时下游 step 的 input schema 校验会捕获（如果有 schema 的话）
- 多上游 step 不声明 input_mapping 时，**注册时报错**（POST /api/v1/flows 返回 400）

#### 数据传递（input_mapping）

```yaml
- name: merge
  rune: report-gen
  depends_on: [analyze, translate]
  input_mapping:
    analysis: analyze.output.result
    translation: translate.output.text
    lang: "$input.target_lang"           # $input 引用 Flow 原始输入
```

- 默认：上游 output 整体作为下游 input
- 多上游汇总时必须声明 mapping

#### 调用模式

- **sync**：等所有 step 完成，返回最终 output
- **async**：返回 flow_task_id，可查询进度（当前在哪个 step）
- **stream**：逐 step 推送 SSE 事件（step_started / step_completed / flow_completed）

#### 持久化

- Flow 定义存 SQLite
- Flow 执行记录存 SQLite（每个 step 的输入输出和耗时）

#### API

| 路由 | 方法 | 说明 |
|------|------|------|
| `/api/v1/flows` | POST | 注册 Flow（JSON body；YAML 由 CLI 在本地转 JSON 后调 API） |
| `/api/v1/flows` | GET | 列出所有 Flow |
| `/api/v1/flows/:name` | GET | 获取 Flow 定义 |
| `/api/v1/flows/:name` | DELETE | 删除 Flow |
| `/api/v1/flows/:name/run` | POST | 执行 Flow |

Flow 也可声明 `gate_path`，暴露为业务路由。

### Implementation

- 重写 `runtime/rune-flow/`
- DAG 拓扑排序 + 层级并行执行
- 条件表达式使用 `cel-rust`（Common Expression Language），语法安全且沙箱化，适合配置驱动的条件判断
- Flow step 复用 `RuneInvoker` 接口

## Module 6: TypeScript SDK

### Design

正式 npm 包 `@rune-sdk/caster`，与 Python SDK 能力对齐。

#### 核心能力

- gRPC 双向流（`@grpc/grpc-js`）
- Caster 类 + 方法注册 Rune
- unary handler + stream handler
- 心跳、cancel 感知（通过 `AbortSignal` 暴露 cancel 状态）、指数退避重连
- 文件接收支持
- TypeScript 类型完备
- stream handler 中 `AbortSignal` abort 时自动清理资源

#### API 风格

```typescript
const caster = new Caster({
  runtime: "localhost:50070",
  key: "rk_xxx"
});

caster.rune({
  name: "translate",
  gate: { path: "/translate" },
  inputSchema: { type: "object", properties: { text: { type: "string" } } },
  supportsStream: true,
}, async (ctx, input) => {
  return { translated: await doTranslate(input.text) };
});

caster.streamRune({
  name: "chat",
  gate: { path: "/chat" },
}, async (ctx, input, stream) => {
  for await (const token of llm.stream(input.prompt)) {
    await stream.emit(token);
  }
});

await caster.run();
```

#### 包结构

```
sdks/typescript/
├── src/
│   ├── caster.ts
│   ├── handler.ts
│   ├── stream.ts
│   ├── types.ts
│   └── proto/           # protobuf 生成代码（内嵌）
├── package.json
├── tsconfig.json
└── tests/
```

### Implementation

- proto 代码通过 `buf` 或 `grpc-tools` 生成，内嵌在包里
- 用户不需要自己编译 proto
- 先本地 link 自用，稳定后发布到 npmjs.com

## Module 7: CLI Tool

### Design

独立的 `rune-cli` 二进制，通过 HTTP API 与 Runtime 通信。

```
Runtime 管理:
  rune start                    # 启动 Runtime（前台）
  rune start -d                 # 后台守护进程模式
  rune stop                     # 停止后台 Runtime
  rune status                   # 运行状态

Key 管理:
  rune key create --type caster --label "my-mac"
  rune key create --type gate --label "game-project"
  rune key list
  rune key revoke <key_id>

Rune 查看与调用:
  rune list                     # 列出所有在线 Rune
  rune info <name>              # 详情（schema、Caster 列表）
  rune call <name> '{...}'      # 同步调用
  rune call <name> --stream     # 流式调用
  rune call <name> --async      # 异步调用
  rune task <task_id>           # 查询异步任务

Flow 管理:
  rune flow register flow.yaml
  rune flow list
  rune flow info <name>
  rune flow run <name> '{...}'
  rune flow delete <name>

日志与统计:
  rune logs                     # 最近调用日志
  rune logs --rune pdf-parse    # 按 Rune 筛选
  rune logs --since 1h          # 按时间筛选
  rune stats                    # 调用统计

配置:
  rune config init              # 生成默认配置
  rune config show              # 显示当前配置

远程管理:
  rune --remote https://rune.chasey.ai status
```

### 守护进程模式

- PID 文件：`~/.rune/rune.pid`
- `rune stop` 读 PID 文件发送 SIGTERM
- 不做自动重启，生产环境建议用 systemd/launchd 管理
- `rune status` 检查 PID 文件 + 进程存活

### Implementation

- 和 `rune-server` 共用 Cargo workspace
- 新增 `runtime/rune-cli/` crate
- 使用 `clap` crate 做命令解析
- 通过 HTTP API 通信，不直接操作内部状态

## Module 8: Configuration System

### Design

TOML 配置文件，统一管理 Runtime 所有参数。

```toml
[server]
grpc_host = "0.0.0.0"
grpc_port = 50070
http_host = "0.0.0.0"
http_port = 50060
dev_mode = false
drain_timeout_secs = 30

[auth]
enabled = true
exempt_routes = ["/health"]

[store]
db_path = "~/.rune/rune.db"
log_retention_days = 30

[session]
heartbeat_interval_secs = 10
heartbeat_timeout_secs = 35
max_request_timeout_secs = 300

[gate]
cors_origins = ["*"]
max_upload_size_mb = 50

[resolver]
strategy = "round_robin"

[rate_limit]
requests_per_minute = 600

[log]
level = "info"
file = "~/.rune/logs/rune.log"    # 可选，不配则仅 stdout
```

### Priority

CLI 参数 > 环境变量（`RUNE_*`） > 配置文件 > 默认值

### Implementation

- 使用 `toml` + `serde` crate
- `rune config init` 生成带注释的默认配置
- `rune start --dev` 自动覆盖：`dev_mode=true`, `auth.enabled=false`, 绑定 `127.0.0.1`

## Module 9: Observability & Operations

### Design

1. **结构化日志**
   - JSON 格式输出（`tracing-subscriber` JSON layer）
   - 每条包含：timestamp, level, module, request_id, rune_name, caster_id
   - 日志级别可运行时调整：`RUNE_LOG=info`

2. **调用指标**（基于 SQLite call_logs 计算）
   - 每个 Rune：调用次数、成功率、平均耗时、P95 耗时
   - 每个 Caster：处理请求数、当前并发、在线时长
   - 通过 `rune stats` 或 `GET /api/v1/stats` 查看

3. **Runtime 状态 API**
   - `GET /api/v1/status` — uptime、Caster 数、Rune 数、pending tasks、版本
   - `GET /api/v1/casters` — 在线 Caster 详情（runes、负载、连接时间）

4. **Rate Limiting**
   - **Per Gate Key** 的请求频率限制（每个 Key 独立计数）
   - `[rate_limit] requests_per_minute = 600`（每个 Key 每分钟 600 次）
   - 算法：固定窗口（实现简单，个人用足够）
   - 超限返回 `429 Too Many Requests`
   - dev 模式下 Rate Limit 自动禁用

5. **Graceful Shutdown**
   - SIGTERM/SIGINT → draining 状态
   - 停止接受新请求 → 等待进行中请求完成 → 通知 Caster DetachRequest → 超时强制关闭
   - `[server] drain_timeout_secs = 30`

## Module 10: Advanced Scheduling

### Design

1. **内置策略**
   - `round_robin`：轮询（现有）
   - `random`：随机
   - `least_load`：当前并发最低优先

2. **Labels 路由**
   - Caster 注册携带 labels：`labels={"gpu": "a100", "region": "asia"}`
   - 调用时指定：`Header: X-Rune-Labels: gpu=a100`
   - 先按 labels 过滤，再按 strategy 调度
   - 不匹配返回 `503 Service Unavailable`

3. **优先级队列**
   - Caster 声明 `priority`（默认 0，越大越优先）
   - 场景：A100 `priority=10`，Mac `priority=1` → 有 A100 时优先用，下线后 fallback

## Module 11: Protocol Evolution

### Proto Changes

```protobuf
message RuneDeclaration {
    string name = 1;
    string version = 2;
    string description = 3;
    string input_schema = 4;         // 恢复：JSON Schema string
    string output_schema = 5;        // 恢复：JSON Schema string
    bool supports_stream = 6;
    GateConfig gate = 7;
    int32 priority = 8;              // 新增：Caster 优先级
}

message FileAttachment {             // 新增
    string filename = 1;
    bytes data = 2;
    string mime_type = 3;
}

message ExecuteRequest {
    string request_id = 1;
    string rune_name = 2;
    bytes input = 3;
    map<string, string> context = 4;
    uint64 timeout_ms = 5;
    repeated FileAttachment attachments = 6;  // 新增
}

message ExecuteResult {
    string request_id = 1;
    bytes output = 2;
    string error = 3;
    repeated FileAttachment attachments = 4;  // 新增
}

message AttachAck {
    bool accepted = 1;
    string reason = 2;
    repeated string supported_features = 3;   // 新增：特性协商
}
```

### New Protocol Guarantees

追加到现有 13 条之后：

14. **Schema gate**: Schema 校验失败在 Gate 层拦截，不到达 Caster
15. **Upload gate**: 文件附件大小超限在 Gate 层拦截
16. **Label strictness**: Labels 不匹配时返回 Unavailable，不降级到无 label Caster
17. **Flow failure isolation**: Flow step 失败后，未启动的下游 step 不会执行
18. **Graceful drain**: Graceful shutdown 期间，进行中请求保证完成或超时

### Compatibility

- 包名保持 `rune.wire.v1`（字段级向后兼容）
- `input_schema`（字段号 4）和 `output_schema`（字段号 5）是在 v0.1.0 reserved 的相同编号上恢复相同类型（string）的字段，符合 proto3 向后兼容规则，v0.1.x SDK 会忽略这两个字段
- 新字段全部 optional，老 SDK 不受影响
- Runtime 通过 `AttachAck.supported_features` 告知 Caster 支持的特性集

## Non-Goals (v1.0.0)

以下能力明确不在 v1.0.0 范围内：

# Rune v0.1.0 Scope

> 综合 Claude draft、Codex draft、Codex review feedback 产出的终版。

## Context

Rune 已完成 4 轮 POC（v1-v4），验证了协议桥接、本地/远程统一调用、并发会话、SSE、异步任务和最小链式 Flow。多轮 review 暴露出一个共同结论：

- Rune 的方向是对的。
- Rune 还不能以当前 POC 形态直接作为正式框架发布。
- 真正阻碍发布的不是功能不够多，而是 correctness 和承诺闭环还不够强。

v0.1.0 的目标不是做成"功能丰富版"，而是做成第一个**正确、可交付、可验证**的正式版本。

> Rune v0.1.0 是一个协议驱动的多语言函数执行框架的首个正式版本，提供 Rust 参考 runtime、官方 Python Caster SDK，以及正确的 sync / real stream / async / chain flow 语义。

**第一原则：correctness > capability breadth。**

---

## Product Positioning

### Official Positioning

Rune 是一个**协议优先的多语言框架**。核心产出：

1. **Rune Wire Protocol (RWP)** — 唯一正式协议源
2. **Rust Runtime** — 官方参考服务端实现
3. **多语言 Caster SDK** — 各语言原生实现，先从 Python 开始

### Runtime First

v0.1.0 采用 `Runtime First` 路线。正式宣传的能力只有：

- 定义 Rune 并自动暴露真实 HTTP 路由
- 本地执行与远程 Caster 执行统一
- sync / real stream / async 三种调用模式
- 最小顺序 Flow
- 官方 Python SDK

v0.1.0 **不再宣传**：

- Schema 校验 / OpenAPI 生成
- 完整 DAG 工作流
- Durable execution
- 多语言 SDK 同步成熟
- 完整 API contract 平台

README、官网和示例文案必须同步修订，只承诺上述真实能力。

### Success Criteria

以下用户故事必须真实成立：

1. 用户 `pip install rune-sdk`
2. 用户注册一个带 `gate="/translate"` 的 Python Rune
3. `POST /translate` 正常返回
4. `POST /translate?stream=true` 走真实流式，不是"先执行完再切片"
5. `POST /translate?async=true` 返回 task id，后续可查询结果
6. 本地 Rune、远程 Rune、Flow step 的错误与超时语义一致

### Structure Decisions

| 决策 | 理由 |
|------|------|
| Monorepo | proto 是跨语言契约，必须和 runtime / SDK 同步演进 |
| `runtime/` + `sdks/` 并列 | Rune 不是"纯 Rust 项目"，明确服务端 vs 客户端 SDK 分区 |
| `proto/` 在根目录 | 协议不从属于任何一个实现 |
| `demo/` 只读保留 | POC 历史有参考价值 |
| 未来 SDK 占位 | 创建占位 README，不进入 release promise |

### Release Promise vs Repo Layout

目录存在不等于版本承诺：

- **Stable in 0.1.0:** Rust runtime, Python SDK
- **Not in release promise:** Go SDK, TypeScript SDK, Rust SDK

## Scope

### 1. Must-Fix Correctness Issues

v0.1.0 在新增功能前，必须先修完以下 correctness 问题：

#### 1.1 Permit 双重归还

**位置：** `session.rs:141-145`（Result），`:170-174`（StreamEnd）

**机制：** `add_permits(1)` 无条件执行，不管 `pending.remove()` 是否成功。超时已归还的 permit 被迟到 Result 再归还一次，semaphore 容量膨胀，`max_concurrent` 失效。

**修复：** `add_permits(1)` 移入 `pending.remove()` 的 `Some(_)` 分支。迟到的 Result/StreamEnd 静默丢弃。

#### 1.2 假流式

**位置：** `gate.rs:121-151`

**机制：** `stream=true` 分支调用 `invoker.invoke()` 拿完整结果后按 20 字符切片发 SSE。`RuneInvoker` 没有 stream 方法。

**修复：** `RuneInvoker` 增加 `invoke_stream()`，Gate SSE 桥接 `mpsc::Receiver` 到 event stream。

#### 1.3 gate_path 悬空

**位置：** `gate.rs:41-49` 硬编码路由

**机制：** `gate_path` 只被存储和展示，从未用于路由生成。

**修复：** Gate 通过 fallback handler + Relay 查表实现动态路由。

#### 1.4 心跳不强制驱逐

**位置：** `session.rs:64-76`

**修复：** 心跳超时后主动 drop outbound channel，触发完整断线清理。

#### 1.5 cancel() 无上游入口

`cancel()` 存在但无调用者。Gate 增加 `DELETE /api/v1/tasks/:id`。

#### 1.6 Session/Gate 零测试

session 并发语义、gate HTTP 行为完全没有自动化约束。

### 2. Protocol Scope

v0.1.0 固定使用 `rune.wire.v1`。

#### Officially Supported Messages

CasterAttach / AttachAck / CasterDetach / ExecuteRequest / ExecuteResult / StreamEvent / StreamEnd / CancelRequest / Heartbeat

#### Officially Supported Declaration Fields

| 字段 | runtime 行为 |
|------|-------------|
| `RuneDeclaration.name` | 注册到 Relay，用于路由 |
| `RuneDeclaration.version` | 存储到 RuneConfig |
| `RuneDeclaration.description` | 存储到 RuneConfig |
| `RuneDeclaration.supports_stream` | Gate 据此允许/拒绝 `?stream=true` |
| `RuneDeclaration.gate.path` | Gate 据此生成业务 HTTP 路由 |
| `RuneDeclaration.gate.method` | Gate 据此决定 HTTP 方法 |
| `CasterAttach.max_concurrent` | Semaphore 初始化容量 |
| `CasterAttach.labels` | 仅作透明元数据保留，不进入调度语义 |
| `ExecuteRequest.context` | 透传到 RuneContext.context |
| `ExecuteRequest.timeout_ms` | 作为 per-request timeout |

**原则：proto 中保留的字段，runtime 必须真实保留和解释。不允许再出现"proto 有、实现忽略"的情况。**

#### Schema 字段处理

`input_schema` 和 `output_schema` 不属于 v0.1.0 正式能力。**从 v0.1.0 的稳定 proto 中移除 schema 字段**，后续版本真做 contract 时再新增。字段编号（4, 5）保留不复用。

### 3. Runtime Core Scope

#### Public Types

```rust
pub struct RuneConfig {
    pub name: String,
    pub version: String,
    pub description: String,
    pub supports_stream: bool,
    pub gate: Option<GateConfig>,
}

pub struct GateConfig {
    pub path: String,
    pub method: String,  // 默认 "POST"
}

pub struct RuneContext {
    pub rune_name: String,
    pub request_id: String,
    pub context: HashMap<String, String>,
    pub timeout: Duration,
}
```

#### Core Interfaces

```rust
// unary handler
pub type RuneHandler = Arc<
    dyn Fn(RuneContext, Bytes) -> Pin<Box<dyn Future<Output = Result<Bytes, RuneError>> + Send>>
        + Send + Sync,
>;

// streaming handler
#[async_trait]
pub trait StreamRuneHandler: Send + Sync + 'static {
    async fn execute(&self, ctx: RuneContext, input: Bytes, tx: StreamSender)
        -> Result<(), RuneError>;
}

pub struct StreamSender { /* mpsc::Sender */ }
impl StreamSender {
    pub async fn emit(&self, data: Bytes) -> Result<(), RuneError>;
    pub async fn end(self) -> Result<(), RuneError>;
}

// 统一调用接口
#[async_trait]
pub trait RuneInvoker: Send + Sync {
    async fn invoke_once(&self, ctx: RuneContext, input: Bytes) -> Result<Bytes, RuneError>;
    async fn invoke_stream(&self, ctx: RuneContext, input: Bytes)
        -> Result<mpsc::Receiver<Result<Bytes, RuneError>>, RuneError>;
}

// 纯注册表
pub struct Relay { entries: DashMap<String, Vec<RuneEntry>> }

// 调度策略
pub trait Resolver: Send + Sync {
    fn pick<'a>(&self, candidates: &'a [RuneEntry]) -> Option<&'a RuneEntry>;
}

// 最小生命周期
pub struct App { /* relay, resolver, session_mgr, config */ }
impl App {
    pub fn new() -> Self;
    pub fn rune(&mut self, config: RuneConfig, handler: RuneHandler) -> &mut Self;
    pub fn stream_rune(&mut self, config: RuneConfig, handler: impl StreamRuneHandler) -> &mut Self;
    pub fn set_resolver(&mut self, r: impl Resolver + 'static) -> &mut Self;
    pub async fn run(&mut self) -> Result<(), anyhow::Error>;
}
```

#### Required Runtime Behaviors

- local Rune 和 remote Rune 走统一调用抽象
- local stream 和 remote stream 语义一致
- `max_concurrent` 严格生效
- timeout / cancel / disconnect / late result 状态收敛
- flow step 复用同一套 invoker 语义
- permit 归还必须与 `pending.remove()` 成功绑定

#### Session State Machine

v0.1.0 最小显式状态机：`Attaching` → `Active` → `Disconnected`

`Draining` 和 `drain_timeout` 不进入 v0.1.0。

### 4. Gate / HTTP Scope

#### Official Routes

| 路由 | 方法 | 说明 |
|------|------|------|
| `/health` | GET | 健康检查 |
| `/api/v1/runes` | GET | 列出所有注册 Rune |
| `/api/v1/runes/:name/run` | POST | 调试入口（不是主业务路由） |
| `/api/v1/tasks/:id` | GET | 查询异步任务 |
| `/api/v1/tasks/:id` | DELETE | 取消异步任务 |
| 业务 Rune 路由 | 按 gate.method | 按 `gate.path` + `gate.method` 自动生成 |

Flow 路由（`/api/v1/flows`、`/api/v1/flows/:name/run`）由 rune-flow 通过 Gate 的 route registration API 自行挂载。Gate crate 不直接依赖 rune-flow crate。

#### gate.path 冲突策略

| 场景 | 处理 |
|------|------|
| 同一 `rune_name` 的多个 Caster 实例声明相同 `gate.path` | **允许**。共享同一路由，Resolver 在实例间轮询 |
| 不同 `rune_name` 声明相同 `gate.path + gate.method` | **硬错误**。Attach 时检测冲突，返回 `AttachAck { accepted: false, reason: "route conflict" }`。本地注册时 panic |
| Caster 声明的 `gate.path` 与管理路由（`/health`、`/api/v1/*`）冲突 | **硬错误**。同上 |

#### Required Gate Behaviors

- 声明了 `gate.path=/echo` 的 Rune 必须可以通过 `/echo` 调用
- `stream=true` 必须走 `invoke_stream()`，直接桥接到 SSE
- `supports_stream=false` 的 Rune 请求 stream 时返回 `400 Bad Request`
- `async=true` 返回 task id，可查询结果
- 未声明 `gate.path` 的 Rune 不暴露业务路由（只能通过调试入口 `/api/v1/runes/:name/run` 调用）
- sync / stream / async 共享同一套错误码映射

### 5. Async / Task Scope

最小正式版，不是 durable job system。

#### Task States

`pending` → `running` → `completed` / `failed` / `cancelled`

#### Cancel 语义（固定规则）

| 规则 | 说明 |
|------|------|
| task_id = request_id | 一一对应，不额外生成 |
| `DELETE /api/v1/tasks/:id` 幂等 | 重复 DELETE 返回 200，不报错 |
| 已完成任务 DELETE | 返回 `409 Conflict`，body 说明任务已终结 |
| SSE 客户端断开 = cancel | runtime 检测 SSE 连接关闭后，对底层 request 发起 cancel；Caster 侧感知 cancel 后停止执行 |
| `DELETE` 只适用于 async task | sync 请求不提供 cancel API |
| cancel 框架保证 | 请求方收到 cancelled、pending 清理、permit 回收、迟到结果不污染状态 |

#### Implementation Boundary

- task store 为单进程内存（DashMap）
- 不做持久化、重启恢复、分页、搜索、租户隔离

### 6. Flow Scope

只到**最小顺序 Chain**。

#### Required

- Flow 注册 / 列表 / sync 执行 / async 执行
- local + remote mixed chain
- step-level error reporting（step index + rune name）
- step request_id 从 flow request 派生，可追踪

#### Not in scope

并行 DAG、分支、循环、子流程、YAML/JSON flow 加载、flow streaming、compensation/saga、flow persistence。

### 7. Python SDK Scope

**最小正式 SDK**，不是高阶 DX 版本。

#### Required

- connect / attach / re-attach（指数退避重连）
- `@caster.rune()` / `@caster.stream_rune()` 装饰器
- heartbeat / once 执行 / stream 执行
- cancel 感知
- grpc.aio（asyncio 原生）
- type hints
- pip-installable

#### Not in scope

typed schema、codegen、高级 decorator ergonomics、插件系统、复杂配置。

## Guarantees

v0.1.0 必须对外保证：

1. 本地 Rune 和远程 Rune 在调用语义上统一
2. sync / stream / async 是同一 Rune 的三种正式模式，不是三套实现
3. stream 首事件来自真实执行过程
4. `gate.path` 是真实业务路由，不是 metadata
5. timeout / cancel / disconnect 后，runtime 状态必须收敛
6. late result / late stream end 不得污染容量和任务状态
7. Python Caster 可参与完整正式链路
8. mixed local/remote flow 可成功运行

- attach 成功注册
- attach 后 metadata 正确保留
- heartbeat 正常续命
- heartbeat 超时触发驱逐
- detach 正常清理
- disconnect 清理 pending once
- disconnect 清理 pending stream
- timeout 结束 once 请求
- timeout 结束 stream 请求
- cancel 结束 once 请求

- late result 被忽略且不重复归还 permit

- 已完成任务 DELETE 返回 409

- 不同 rune_name 声明相同 gate.path 时 attach 失败

### 3. Flow

- local chain success
- remote chain success
- mixed chain success
- step not found
- step failed
- empty flow
- async flow success
- flow task 查询成功

### 4. Python SDK E2E

- Python unary Rune attach + invoke
- Python stream Rune attach + invoke
- Python slow Rune async
- Python task cancel
- Python caster disconnect + reconnect
- Python steps in mixed flow

### 5. Regression

专门锁死 POC 阶段暴露过的问题：

| 回归用例 | 锁定的 bug |
|---------|-----------|
| fake stream regression | stream=true 不能先执行完再切片 |
| gate_path display-only regression | gate_path 必须生成真实 HTTP 路由 |
| double permit release regression | 迟到 Result 不能膨胀 semaphore |
| heartbeat timeout not evicting regression | 心跳超时必须触发完整清理 |
| proto metadata dropped regression | 声明的字段 runtime 必须使用 |

## Default Assumptions

- v0.1.0 采用 Runtime First 路线
- Python SDK 纳入 v0.1.0
- Schema / OpenAPI 不纳入 v0.1.0
- Schema 字段从稳定 proto 中移除

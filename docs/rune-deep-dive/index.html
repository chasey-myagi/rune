Rune：定义函数，获得一切

协议优先的分布式函数执行框架。写一个函数，自动获得 HTTP 端点、流式推送、异步任务、DAG 编排、负载均衡 — 全部开箱即用，零胶水代码。

01

Rune 到底是什么？

一句话概括：你写函数，框架替你搞定其余所有分布式执行的事情。

想象你有一个 Python 函数做翻译、一个 TypeScript 函数做图片生成、一个 Rust 函数做数据处理。传统方式下，你需要为每个函数搭建 HTTP 服务、处理负载均衡、实现流式推送、管理异步任务……这些「胶水代码」的工作量往往是业务逻辑的十倍。

Rune 的做法：把函数的「执行」和「暴露」彻底分离。函数作者（Caster）只关心业务逻辑，通过统一的 Wire Protocol 向中央 Runtime 注册。Runtime 自动完成 HTTP 路由、调度、并发背压、工作流编排。

一个最小的例子

用 Python 写一个翻译函数，注册到 Rune：

translate_caster.py 
from rune_sdk import Caster

caster = Caster("localhost:50070")

@caster.rune("translate", gate="/translate")
async def translate(ctx, input):
    data = json.loads(input)
    result = do_translate(data["text"], data["lang"])
    return json.dumps({"translated": result})

caster.run()
# 自动获得 ↓
# POST /translate           → 同步调用
# POST /translate?stream    → SSE 流式推送
# POST /translate?async     → 异步任务（返回 task_id）
# + 负载均衡 / 心跳检测 / 自动重连 / 并发控制 ...
02

解决了什么问题？

Rune 瞄准的是一个反复出现的痛点：函数到服务之间巨大的「基础设施鸿沟」。

传统方式的五大痛点

胶水代码爆炸
写一个函数 20 行，为它搭 HTTP 服务 200 行。加上错误处理、超时、重试、日志、监控… 胶水代码是业务代码的 10 倍，且每个函数都要重复。
多语言团队的协作地狱
AI 团队用 Python，前端用 TypeScript，基础设施用 Rust/Go。每种语言各搞一套 HTTP 框架、部署流程、监控体系，互相调用还要对齐协议。
流式和异步是二等公民
同步调用很简单，但加 SSE 流式？重写 handler。加异步任务？再搭一套消息队列 + 状态管理。三种模式三套代码，三倍维护成本。
编排靠硬编码
「先调 A，结果传 B，B 和 C 并行，根据 B 的输出决定是否调 D」— 全靠 if/else 和 Promise.all 硬写，出错无法追踪，改动牵一发动全身。
状态管理的黑洞
函数执行一半超时了、连接断了、请求取消了 — 信号量归还了吗？并发计数减了吗？资源释放了吗？大多数框架的答案是「不保证」。
核心洞察：90% 的分布式函数执行需求都遵循同一模式 — 注册函数、路由请求、管理连接、控制并发。Rune 把这个模式固化为一套协议和运行时，让你永远只需要写那 20 行业务逻辑。

四层架构

0
协议层
rune.proto · Rune Wire Protocol
一份 Protocol Buffer 定义 Caster 与 Runtime 的所有交互。语言无关 — 任何实现 RWP 的客户端都可以接入。
1
运行时核心
rune-core · ~3000 行 Rust
Relay（注册表：谁在哪）、Resolver（调度：选谁执行）、Invoker（统一调用：屏蔽本地/远程差异）、SessionManager（连接管理：心跳/并发/重连）。
2
HTTP 网关（可选）
rune-gate · ~1500 行 Rust · Axum
自动路由映射、sync/stream(SSE)/async 三模式、Schema 校验、OpenAPI 生成、文件传输。Gate 是一层「皮肤」，不是核心 — 纯 gRPC 场景可以不用它。
3
扩展层
rune-flow · rune-store · rune-schema
DAG 工作流引擎、SQLite 持久化、JSON Schema 校验。独立 crate，按需引入。Flow 复用同一个 Invoker 接口，函数被直接调用还是作为 Flow 步骤，行为完全一致。

核心设计决策

决策为什么
gRPC 双向流Runtime 需要主动向 Caster 推送请求、取消、心跳。HTTP 是单向的，WebSocket 没有类型安全和代码生成。
bytes 不是 string热路径省掉 UTF-8 校验，不绑定 JSON。未来可以传 MessagePack / CBOR / Protobuf。
Invoker 统一本地/远程上层不关心函数在哪。重试、超时、metrics 在 Invoker 上包装，一次加全部受益。
Relay ≠ Resolver数据和策略分离。更换调度算法不影响注册逻辑，反之亦然。
协议保证是合约18 条行为保证写入文档并用回归测试锁定。违反即 bug，不可商量。
04

Rune Wire Protocol：协议即灵魂

Rune 自称「协议优先」。整个框架的核心不是 Runtime 的 Rust 代码，而是那份 .proto 文件定义的 Rune Wire Protocol (RWP)。所有 SDK 只需实现这个协议，就能接入任意语言。

为什么是双向流？

传统 RPC 是「Client 请求，Server 响应」。但 Rune 的场景不同 — Runtime 需要主动向 Caster 推送执行请求、取消指令、心跳检测。两端都能主动说话，天然适合 gRPC 双向流。

状态收敛保证

协议设计中最精妙的部分是状态收敛：无论发生什么异常（超时、取消、断连、迟到响应），Runtime 的内部状态总是最终归于一致。

保证 #7 — No State Pollution：迟到的 Result 或 StreamEnd（在超时/取消/断连已清理完状态之后到达的）被静默忽略。不会返回幽灵信号量，不会腐蚀运行时状态。
保证 #8 — Strict max_concurrent：Caster 声明的并发容量通过 Semaphore 严格执行。无论任何错误路径，permit 计数永远不会超过声明值。

Rune vs 已有方案

Rune 不是第一个简化分布式函数执行的框架。但它的定位独特 — 融合 Serverless、RPC、工作流引擎三者的核心能力于一个协议。

关键差异：Rune 不是 Serverless，不是 RPC 框架，不是工作流引擎 — 它把三者的核心能力融合在一个协议里。你不需要 API Gateway + Lambda + Step Functions + CloudWatch 四件套。一个 Runtime，全包。

得天独厚的场景

在这些场景下，Rune 相比其他方案有结构性优势。

🧠
AI/ML 模型推理服务化
AI 团队用 Python 写推理代码，Rune 自动给出 HTTP 端点。流式输出天然适配 LLM 逐 token 生成。GPU Worker 声明 max_concurrent=2 做背压，标签路由 gpu=a100 实现硬件感知调度。零冷启动，不改一行推理代码。
流式推送GPU 调度背压控制零冷启动
🔗
多语言微服务统一入口
Python 写 AI 能力、TypeScript 写业务逻辑、Rust 写性能热路径 — 三种语言通过同一个 Wire Protocol 注册到同一个 Runtime，对外暴露统一 HTTP API。不再为每种语言维护独立的框架和部署流程。
多语言统一统一入口协议标准化
🔀
复杂业务管线编排
用 DAG Flow 编排多步骤管线：文档解析 → 翻译和摘要并行 → 条件分支决定是否人工校对 → 存储。Flow 的 input_mapping 自动串联上下游数据，单步失败不影响已完成分支的结果。
DAG 编排并行执行条件分支故障隔离
⚡
内部工具 / 能力平台
数据导出、邮件发送、报表生成 — 注册为 Rune，统一通过 CLI 或 HTTP 调用。API Key 双层隔离，Rate Limiting 防滥用，调用日志和统计开箱即用。
认证隔离频率限制调用审计CLI
📡
实时流式处理
处理函数通过 StreamSender 实时推送中间结果，前端通过 SSE 实时接收。实时日志分析、流式报告生成、逐行代码审查 — 这是真流式，第一个事件在 handler 产生输出时立即发出。
SSE 流式真实时低延迟
🏗️
长时间异步任务
视频转码、大文件处理、批量迁移 — 以 ?async 提交后立即返回 task_id，随时轮询状态，结果持久化到 SQLite。Caster 断连重连后任务不丢失。
异步任务持久化断点续传

边界在哪？

清醒认识边界，比盲目吹捧更有价值。

适合

  • 函数级别的分布式执行
  • 多语言团队统一协作
  • sync + stream + async 多模式
  • GPU / 长任务 / AI 推理
  • 中小规模 DAG 编排
  • 快速把函数变成 HTTP API
  • 自托管、无供应商锁定

不适合

  • 超大规模集群编排（Kubernetes）
  • 事务性长工作流（Temporal）
  • 纯 RPC 通信（gRPC 直连更轻量）
  • 事件驱动（Kafka / RabbitMQ）
  • 极端低延迟（多一跳中转）
  • 弹性伸缩（Serverless 更成熟）
  • 已有服务网格的大型组织
经验法则：2–50 人团队，2–3 种语言，需要快速把能力服务化并编排 — Rune 是最小阻力路径。千节点集群或 exactly-once 事务 — 你需要更重的基础设施。
08

结论

Rune = 函数到服务的最短路径

核心价值：消除从「写了一个函数」到「它成为可路由、可监控、可编排的生产服务」之间的全部摩擦。

独特之处不在于某个单一能力 — 流式、编排、负载均衡别人也有 — 而在于这些能力全部融合在一个统一协议和运行时里，让函数作者只需关心 20 行业务逻辑。

多语言团队、AI/ML 服务化、内部能力平台 — 这些场景下，Rune 提供「写函数就行，其余全包」的解决方案。以前你需要 API Gateway + RPC 框架 + 工作流引擎 + 监控系统四件套，现在一个 Runtime 搞定。

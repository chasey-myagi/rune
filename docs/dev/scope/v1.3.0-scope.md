# Rune v1.3.0 Scope

> Advanced Flow Control — 从 DAG 到完整工作流原语

**前置版本**: v1.2.0  
**核心目标**: 在 rune-flow 中引入高级 Step 类型，使 Flow 能够表达迭代、数据并行、值路由、子流组合等完整的工作流控制原语，同时保持对现有 DAG Flow 的完全向后兼容。

---

## 背景：Jig Harness 驱动需求

v1.3.0 的功能设计来自两个方向：Rune 自身的工作流表达能力演进，以及 Jig Harness 作为第一个重度使用方暴露的具体缺口。

| 模块 | Jig Harness 用途 |
|------|-----------------|
| Loop (P0) | impl-review / tdd / frontend-design 的质量门控循环 |
| Switch (P1) | 按 task type 路由到不同 Flow |
| Flow Step (P1) | commit + create_pr 提成公共子 Flow，多种 Harness 复用 |
| Retry (P2) | create_pr 调 gh CLI 失败时重试 |
| BodyStep name+condition (P0) | TDD loop body 的条件分叉：fix 只在测试失败时运行 |
| Loop SSE 事件 (P1) | jig task logs 实时展示迭代进度 |
| Per-step timeout (P1) | plan 60s / implement 20min / run_tests 5min 差异巨大 |

## 版本总览

### 模块矩阵

| 优先级 | 模块 | 名称 | 核心改动 | 预估工时 |
|--------|------|------|----------|----------|
| P0 | A | Loop Step | `LoopConfig` + 带 name/condition 的 `BodyStep` + 执行引擎循环逻辑 | 5-7d |
| P0 | B | Map Step | `MapConfig` + fan-out/gather 并发执行 | 3-5d |
| P1 | C | Switch Step | `SwitchConfig` + 值路由分支执行 | 2-3d |
| P1 | D | Flow Step | `FlowConfig` + 子流调用 + 循环引用检测 | 2-3d |
| P1 | F | Loop 迭代进度事件 | `FlowProgressEvent` + SSE 流集成 | 3-4d |
| P1 | G | Per-step timeout | `timeout_ms` 字段，覆盖全局默认 | 1-2d |
| P2 | E | Retry 属性 | `RetryConfig` 通用属性，适用于所有 Step 类型 | 2-3d |

**总预估工时**: 18-27 天（单人）/ 8-10 天（3 人并行）

### 推荐实施顺序

```
阶段 1 — P0 核心迭代原语
├── 模块 G：Per-step timeout   ← 改动小，先做，其他模块直接享用
├── 模块 A：Loop Step          ← 建立带 name/condition 的 BodyStep 体系，其他模块依赖它
└── 模块 B：Map Step           ← 依赖 BodyStep 体系（A 完成后立刻做）

阶段 2 — P1 路由、组合与可观测性（可并行）
├── 模块 C：Switch Step        ← 依赖 BodyStep 体系
├── 模块 D：Flow Step          ← 相对独立，最简单
└── 模块 F：Loop 迭代进度事件  ← 依赖 Loop Step（A 完成后可做）

阶段 3 — P2 可靠性增强
└── 模块 E：Retry 属性        ← 与其他模块无依赖，最后叠加
```

> 模块 A 是整个 v1.3.0 的基础。`BodyStep` 升级为"有名字、有条件"的完整公民后，B/C/F 直接复用这套体系。模块 G 最轻量，优先完成可减少其他模块的重复考虑。

## 数据模型设计

### 核心思路

现有 `StepDefinition` 只支持一种执行原语（调用 Rune）。v1.3.0 引入 `type` 字段作为判别符，将执行行为参数化到对应的 Config 结构中。

`BodyStep`（出现在 Loop/Map/Switch body 中）与顶层 `StepDefinition` 对齐，同样拥有 `name`、`condition`、`input_mapping`、`timeout_ms`，成为完整的执行公民，而非盲目的顺序链节点。

**向后兼容**：未带 `type` 字段的 Step 默认视为 `"rune"` 类型，现有 Flow 定义零修改。

### Rust 类型结构

```rust
// dag.rs — 修改 StepDefinition，新增所有配置类型

/// 顶层步骤类型判别符
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "lowercase")]
pub enum StepKind {
    Rune(RuneConfig),
    Loop(LoopConfig),
    Map(MapConfig),
    Switch(SwitchConfig),
    Flow(FlowConfig),
}

/// 修改后的 StepDefinition（外壳字段不变，新增 timeout_ms、retry）
pub struct StepDefinition {
    pub name: String,
    #[serde(default)]
    pub depends_on: Vec<String>,
    pub condition: Option<String>,
    pub input_mapping: Option<HashMap<String, String>>,
    pub timeout_ms: Option<u64>,         // 模块 G：覆盖全局 step_timeout
    pub retry: Option<RetryConfig>,      // 模块 E
    #[serde(flatten)]
    pub kind: StepKind,
}

// ── BodyStep：Loop/Map/Switch body 中的步骤，与顶层对齐 ──

/// body 中步骤的完整定义（有名字、有条件、有 input_mapping）
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct BodyStep {
    pub name: String,                                        // 必填，用于引用和日志
    pub condition: Option<String>,                           // 可选，不满足则 Skip
    pub input_mapping: Option<HashMap<String, String>>,      // 显式数据引用
    pub timeout_ms: Option<u64>,
    pub retry: Option<RetryConfig>,
    #[serde(flatten)]
    pub kind: BodyStepKind,
}

/// body 中步骤的类型判别符（递归）
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "lowercase")]
pub enum BodyStepKind {
    Rune { rune: String },
    Loop(LoopConfig),
    Map(MapConfig),
    Switch(SwitchConfig),
    Flow(FlowConfig),
}

// ── 各类型配置结构 ──

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RuneConfig {
    pub rune: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct LoopConfig {
    pub body: Vec<BodyStep>,
    pub max_iterations: u32,
    pub until: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MapConfig {
    pub over: String,               // 数组路径：$input.x / steps.A.output.list
    pub body: Box<BodyStep>,        // 对每个元素执行的单元
    pub concurrency: Option<usize>, // 并发上限，None = 全并行
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SwitchConfig {
    pub on: String,
    pub cases: Vec<SwitchCase>,
    pub default: Option<Box<BodyStep>>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SwitchCase {
    pub when: serde_json::Value,
    pub body: BodyStep,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct FlowConfig {
    pub flow: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RetryConfig {
    pub max_attempts: u32,
    pub backoff_ms: u64,
}
```

> **serde flatten + tag 问题**：`#[serde(flatten)]` 与 `#[serde(tag = "type")]` 在 serde_json 中有已知限制。实现时需为 `StepDefinition` 和 `BodyStep` 手动实现 `Deserialize`，先提取 `type` 字段，再按 variant 反序列化剩余字段。这是实现细节，不影响对外 JSON 格式。

## 模块 A — Loop Step

### 功能描述

顺序执行 body 中所有步骤，每步都可带 `condition` 和 `input_mapping`；末轮最后一个非 Skip 步骤的输出作为下一轮的入参，循环直到 `until` 条件满足或达到 `max_iterations`。支持任意深度嵌套。

### BodyStep 升级：有名字、有条件的完整公民

Loop body 中的每个步骤现在等同于一个带名字的"局部 step"：

- `name`：必填，用于 `input_mapping` 引用（`steps.implement.output.session_id`）和进度日志
- `condition`：可选，不满足则 Skip（复用顶层 condition 表达式语法，`steps.X.output.field` 指本轮已执行的 body step 输出）
- `input_mapping`：可选，显式声明从哪个上游 body step 取哪些字段，不再只能拿"上一步的完整输出"

### JSON 表示

**TDD Harness 示例（条件分叉）**：
```json
{
  "name": "tdd_loop",
  "type": "loop",
  "max_iterations": 10,
  "until": "steps.run_tests.output.passed == true",
  "body": [
    { "name": "implement", "type": "rune", "rune": "implement" },
    { "name": "run_tests", "type": "rune", "rune": "run_tests" },
    {
      "name": "fix",
      "type": "rune",
      "rune": "fix",
      "condition": "steps.run_tests.output.passed == false",
      "input_mapping": {
        "session_id": "steps.implement.output.session_id",
        "failures":   "steps.run_tests.output.failures"
      }
    },
    {
      "name": "test_review",
      "type": "rune",
      "rune": "test_review",
      "condition": "steps.run_tests.output.passed == true"
    }
  ]
}
```

**嵌套 Loop（大环 α 包含小环 A/B）**：
```json
{
  "name": "alpha",
  "type": "loop",
  "max_iterations": 100,
  "until": "output.ultimate == true",
  "body": [
    {
      "name": "loop_a",
      "type": "loop",
      "max_iterations": 20,
      "until": "output.a_done == true",
      "body": [
        { "name": "a1", "type": "rune", "rune": "agent_a1" },
        { "name": "a2", "type": "rune", "rune": "agent_a2" }
      ]
    },
    {
      "name": "loop_b",
      "type": "loop",
      "max_iterations": 20,
      "until": "output.b_done == true",
      "body": [
        { "name": "b1", "type": "rune", "rune": "agent_b1" }
      ]
    }
  ]
}
```

### 执行语义

```
execute_loop(config, input):
  current = input
  for i in 0..max_iterations:
    body_outputs: HashMap<String, Bytes> = {}   // 本轮各 step 的输出，跨 step 引用

    for step in body:
      // 评估 condition（引用本轮已完成的 body_outputs）
      if condition defined and not met(condition, body_outputs, current):
        mark step as Skipped; continue

      // 构造 step 输入
      step_input = if step.input_mapping defined:
        build_from_mapping(step.input_mapping, body_outputs, current)
      else if previous non-skipped step exists:
        body_outputs[prev_step.name]
      else:
        current

      // 执行
      result = execute_body_step_kind(step.kind, step_input)
      body_outputs.insert(step.name, result)

    // 本轮的"最终输出"= 最后一个非 Skipped step 的输出
    last = last_non_skipped_output(body_outputs) ?? current

    // 评估 until（引用本轮 body_outputs）
    if until defined and met(until, body_outputs, last):
      current = last; break

    current = last

  return current
```

`until` 和 body step `condition` 中 `steps.X.output.field` 均引用**本轮**的 `body_outputs`；`output` 是本轮最后一个非 Skip step 的输出的快捷引用。

### 验证规则

- `body` 不能为空
- `body[i].name` 在同一 body 中不能重复
- `max_iterations` ≥ 1
- `until` 若存在，必须能通过条件表达式预检
- `body[i].condition` 同上
- 递归 validate 每个 body step 的 kind

### 输出

末轮最后一个非 Skipped body step 的输出。达到 `max_iterations` 时正常返回，不报错。

## 模块 B — Map Step

### 功能描述

将输入数组中的每个元素并发地送入同一个处理单元（Rune、Loop 等），收集所有结果，按原始顺序组成输出数组。

### JSON 表示

**基础用法（数组并行处理）**：
```json
{
  "name": "analyze_docs",
  "type": "map",
  "over": "$input.documents",
  "concurrency": 8,
  "body": { "name": "analyze", "type": "rune", "rune": "doc_analyzer" }
}
```

**body 为 Loop（对每个元素做多轮推理）**：
```json
{
  "name": "deep_analyze",
  "type": "map",
  "over": "steps.fetch.output.items",
  "concurrency": 4,
  "body": {
    "name": "refine_loop",
    "type": "loop",
    "max_iterations": 5,
    "until": "output.confident == true",
    "body": [
      { "name": "refine", "type": "rune", "rune": "refine" }
    ]
  }
}
```

### 执行语义

```
execute_map(config, input):
  items = resolve_path(config.over, input)
  if items is not array: error MapOverNotArray

  semaphore = Semaphore(concurrency ?? items.len())
  tasks = []
  for (i, item) in items.enumerate():
    tasks.push(spawn async {
      acquire semaphore
      result = execute_body_step(config.body, item)
      release semaphore
      (i, result)
    })

  results = gather(tasks)    // fail-fast：任一失败则 Map step 失败
  sort by i
  return JSON array of results
```

### 验证规则

- `over` 不能为空
- `concurrency` 若存在必须 ≥ 1
- 递归 validate `body`

### 输出

JSON 数组 `[result_0, result_1, ..., result_n]`，顺序与输入对应。

## 模块 C — Switch Step

### 功能描述

对表达式求值，按结果精确匹配 `cases`，执行对应 body；无匹配时走 `default`。

### JSON 表示

```json
{
  "name": "dispatch",
  "type": "switch",
  "on": "steps.classify.output.mode",
  "cases": [
    {
      "when": "iterative",
      "body": {
        "name": "iter_loop",
        "type": "loop",
        "max_iterations": 10,
        "until": "output.done == true",
        "body": [{ "name": "agent", "type": "rune", "rune": "iterative_agent" }]
      }
    },
    {
      "when": "batch",
      "body": {
        "name": "batch_map",
        "type": "map",
        "over": "$input.items",
        "body": { "name": "proc", "type": "rune", "rune": "batch_processor" }
      }
    }
  ],
  "default": { "name": "fallback", "type": "rune", "rune": "standard_handler" }
}
```

### 执行语义

```
execute_switch(config, input):
  value = resolve_condition_value(config.on, step_outputs, flow_input)
  for case in config.cases:
    if value == case.when:
      return execute_body_step(case.body, input)
  if config.default is Some:
    return execute_body_step(config.default, input)
  return Skipped
```

- `on` 复用 `resolve_condition_value`
- `when` 精确相等比较（`serde_json::Value`）
- 无匹配且无 `default` → step 状态 `Skipped`

### 验证规则

- `on` 不能为空
- `cases` 不能为空
- 同一 Switch 内 `when` 值不允许重复
- 递归 validate 每个 `case.body` 和 `default`

### 输出

匹配 case 的 body 输出；无匹配无 default 时为 `null`（Skipped）。

## 模块 D — Flow Step

### 功能描述

将另一个已注册的 `FlowDefinition` 作为单个步骤调用，实现 Flow 组合与复用。

### JSON 表示

```json
{ "name": "enrich", "type": "flow", "flow": "data_enrichment_pipeline" }
```

```json
{
  "name": "iter_pipeline",
  "type": "loop",
  "max_iterations": 10,
  "until": "output.quality_score > 90",
  "body": [
    { "name": "refine", "type": "flow", "flow": "refinement_flow" }
  ]
}
```

### 执行语义

```
execute_flow_step(config, input):
  flow = engine.get(config.flow) ?? return Err(FlowStepNotFound)
  check_no_cycle(call_stack, config.flow) ?? return Err(CircularFlowRef)
  return runner.execute_flow(flow, input)
```

- 子 Flow 共享同一个 `FlowRunner`，独立的 `step_outputs`
- 循环引用检测通过调用栈 `Vec<String>` 实现

### 新增错误类型

```rust
pub enum FlowError {
    #[error("flow step references unknown flow: '{0}'")]
    FlowStepNotFound(String),
    #[error("circular flow reference detected: {0}")]
    CircularFlowRef(String),   // 包含调用链，如 "a -> b -> a"
}
```

### 验证规则

- `flow` 不能为空
- 注册时不检测子 Flow 是否存在（允许晚于父 Flow 注册）
- 执行时检测存在性和循环引用

### 输出

子 Flow 的最终输出。

## 模块 F — Loop 迭代进度事件

### 背景

当前 `StreamEvent` 是 Rune 执行内部的流式输出（Caster → Runtime 协议层），粒度在单次 Rune 调用内。Jig 的 `jig task logs` 需要更高层次的进度信息：

```
[tdd_loop] iteration 1/10
  ✓ implement  (34s)
  ✗ run_tests  (12s) — 2 assertions failed
  ✓ fix        (48s)
  ↷ test_review skipped
[tdd_loop] iteration 2/10
  ...
```

### 设计

在 `rune-flow` 层定义 `FlowProgressEvent`，通过 channel 从 `execute_loop` 冒泡到 Gate 的 SSE 流。

```rust
// rune-flow/src/engine.rs

#[derive(Debug, Clone, Serialize)]
#[serde(tag = "event", rename_all = "snake_case")]
pub enum FlowProgressEvent {
    LoopIterationStart {
        step: String,        // Loop step 的 name
        iteration: u32,
        max_iterations: u32,
    },
    BodyStepDone {
        step: String,
        body_step: String,
        duration_ms: u64,
    },
    BodyStepSkipped {
        step: String,
        body_step: String,
    },
    BodyStepFailed {
        step: String,
        body_step: String,
        error: String,
        duration_ms: u64,
    },
    LoopIterationEnd {
        step: String,
        iteration: u32,
        duration_ms: u64,
        until_met: bool,
    },
}
```

```rust
// FlowRunner 新增 progress 发送端（可选）
pub struct FlowRunner {
    relay: Arc<Relay>,
    resolver: Arc<dyn Resolver>,
    step_timeout: Duration,
    progress_tx: Option<mpsc::Sender<FlowProgressEvent>>,  // None = 不发事件
}
```

```rust
// Gate SSE handler 初始化时创建 channel，merge 进 SSE 流
let (progress_tx, mut progress_rx) = mpsc::channel::<FlowProgressEvent>(64);
let runner = engine.runner().with_progress(progress_tx);

// SSE stream = flow_result_stream ∪ progress_event_stream
```

SSE 事件格式：
```
event: flow_progress
data: {"event":"loop_iteration_start","step":"tdd_loop","iteration":1,"max_iterations":10}

event: flow_progress
data: {"event":"body_step_done","step":"tdd_loop","body_step":"implement","duration_ms":34200}

event: flow_progress
data: {"event":"body_step_skipped","step":"tdd_loop","body_step":"test_review"}

event: flow_progress
data: {"event":"loop_iteration_end","step":"tdd_loop","iteration":1,"duration_ms":94800,"until_met":false}
```

### 实现要点

- `execute_loop` 接收 `Option<&mpsc::Sender<FlowProgressEvent>>`，在每轮迭代前后、每个 body step 执行前后 send 事件
- send 失败（receiver dropped）静默忽略，不影响 loop 执行
- 非 SSE 路径（同步/异步调用）不传入 sender，零开销
- Map/Switch 的 body 执行暂不发进度事件（v1.4.0 扩展）

## 模块 G — Per-step timeout_ms

### 背景

`FlowEngine` 当前只有全局 `step_timeout`（默认 30s）。Jig Harness 场景中各 step 执行时长差异极大：

| step | 合理超时 |
|------|---------|
| plan | 60s |
| implement | 1200s（20min） |
| run_tests | 300s（5min） |
| create_pr | 30s |

### 设计

在 `StepDefinition` 和 `BodyStep` 上均新增 `timeout_ms: Option<u64>`，执行时优先使用 step 级别的值：

```rust
let effective_timeout = step
    .timeout_ms
    .map(Duration::from_millis)
    .unwrap_or(runner.step_timeout);
```

- `timeout_ms` 为 `None` 时继承全局 `step_timeout`
- `timeout_ms = 0` 视为无超时（文档注明）
- Loop 的 `timeout_ms` 包裹整个 Loop；单次迭代超时依赖各 body step 自身的 `timeout_ms`

### JSON 表示

```json
{
  "name": "tdd_loop",
  "type": "loop",
  "max_iterations": 10,
  "until": "output.passed == true",
  "timeout_ms": 3600000,
  "body": [
    { "name": "implement", "type": "rune", "rune": "implement", "timeout_ms": 600000 },
    { "name": "run_tests", "type": "rune", "rune": "run_tests", "timeout_ms": 300000 }
  ]
}
```

### 验证规则

无特殊验证（`u64`，任意值合法）。

## 模块 E — Retry 属性

### 功能描述

为任意 Step 类型提供通用重试能力。`retry` 是加在 `StepDefinition` / `BodyStep` 上的可选属性。

### JSON 表示

```json
{
  "name": "create_pr",
  "type": "rune",
  "rune": "create_pr_rune",
  "retry": { "max_attempts": 3, "backoff_ms": 500 }
}
```

### 执行语义

```
for attempt in 1..=max_attempts:
  result = execute_step(step, input)
  if Ok: return Ok
  if attempt < max_attempts: sleep(backoff_ms)
return Err(last_error)
```

- `max_attempts` 含第一次执行
- Loop step 的 retry 包裹整个 Loop（整个循环失败才触发重试）
- `backoff_ms` 当前固定等待，v1.4.0 可扩展 exponential

### 验证规则

- `max_attempts` ≥ 1

## Validation 变更汇总

```
validate_dag(flow):
  // 现有检查不变：重复名、未知依赖、环检测、input_mapping、condition 预检
  // 新增：
  for each step in flow.steps:
    match step.kind:
      Loop(config)   => validate_loop_config(config)
      Map(config)    => validate_map_config(config)
      Switch(config) => validate_switch_config(config)
      Flow(config)   => validate_flow_config(config)
      Rune(_)        => 现有逻辑

validate_loop_config(config):
  - body 非空
  - body[i].name 不重复
  - max_iterations ≥ 1
  - until 条件预检（若存在）
  - body[i].condition 预检（若存在）
  - 递归 validate 每个 body step 的 kind

validate_map_config(config):
  - over 非空
  - concurrency ≥ 1（若存在）
  - 递归 validate body.kind

validate_switch_config(config):
  - on 非空
  - cases 非空
  - when 值不重复
  - 递归 validate case.body.kind 和 default.kind

validate_flow_config(config):
  - flow 名非空（存在性在执行时检测）
```

## engine.rs 变更

`FlowRunner` 的 step 分发扩展：

```rust
match &step_def.kind {
    StepKind::Rune(cfg) => { /* 现有逻辑，使用 effective_timeout */ }
    StepKind::Loop(cfg) => {
        join_set.spawn(async move {
            execute_loop(cfg, step_input, &relay, &resolver, effective_timeout,
                         progress_tx.clone()).await   // 模块 F：传入 progress sender
        });
    }
    StepKind::Map(cfg) => {
        join_set.spawn(async move {
            execute_map(cfg, step_input, &relay, &resolver, effective_timeout).await
        });
    }
    StepKind::Switch(cfg) => {
        join_set.spawn(async move {
            execute_switch(cfg, step_input, &step_outputs_snap, &flow_input_json,
                           &relay, &resolver, effective_timeout).await
        });
    }
    StepKind::Flow(cfg) => {
        join_set.spawn(async move {
            execute_flow_step(cfg, step_input, &runner, &call_stack).await
        });
    }
}
```

各高级 step 的执行函数封装独立，接受 `&Relay`、`&dyn Resolver`、`Duration` 作为参数。

## P0 — 生产必需

### 模块 A：重试与断路器 (Retry & Circuit Breaker)

#### 设计概述

当前 `RemoteInvoker` 直接委托 `SessionManager.execute()` 发送 gRPC 请求到 Caster，如果 Caster 挂了或超时，错误会原样返回给 Gate/Flow 上层，没有任何重试或快速失败保护。生产环境中一个 Caster 临时不可用（网络抖动、重启）不应立刻传导为用户可见错误。

设计核心决策是**使用装饰器/包装器模式**，新建 `RetryInvoker` 结构体包装任意 `Arc<dyn RuneInvoker>`。这样做的好处是：
1. 不改变 `RuneInvoker` trait 签名，保持其简洁性（只有 `invoke_once` 和 `invoke_stream` 两个方法）
2. `LocalInvoker` 和 `RemoteInvoker` 完全不受影响
3. 断路器状态以 per-caster 粒度管理，通过 `caster_id` 隔离
4. 重试逻辑可以在 Relay 的 resolve 阶段注入，也可以在 session 层注入，选择在 Relay 层包装更合适——因为 Relay 知道 caster_id

断路器采用经典三状态模型：`Closed`（正常通过）→ `Open`（快速失败，等待 `reset_timeout`）→ `HalfOpen`（放行一个探针请求，成功则回到 Closed，失败则回到 Open）。每个 caster 独立维护状态，存放在 `CircuitBreakerRegistry`（`DashMap<String, CircuitBreaker>`）中。

重试策略采用指数退避 + full jitter `[0, cap]`，只对**可重试错误**生效（默认 `Unavailable`、`Internal`）。`Timeout` 在 `RetryInvoker` 内部**硬编码为永不重试**（无论 `retryable_errors` 配置），因为 runtime 不会在超时时取消远程 caster 执行，重试会导致 rune 并发双重执行。`InvalidInput`、`NotFound`、`Cancelled` 等客户端错误不重试也不计入断路器。`Timeout`、`Unavailable`、`Internal`、`ExecutionFailed` 等后端错误都计入断路器失败计数。`invoke_stream` 不重试（流式请求在中途失败后状态不可恢复）。

###### Task A.2: 实现断路器核心逻辑

- **新建文件**: `runtime/rune-core/src/circuit_breaker.rs`
- **改动文件**: `runtime/rune-core/src/lib.rs`（第 1-11 行，添加 `pub mod circuit_breaker;`）
- **改动内容**:
  - 断路器状态枚举：
    ```rust
    #[derive(Debug, Clone, Copy, PartialEq)]
    pub enum CBState {
        Closed,
        Open { since: Instant },
        HalfOpen,
    }
    ```
  - `CircuitBreaker` 结构体（单个 caster 的断路器实例）：
    ```rust
    pub struct CircuitBreaker {
        inner: Mutex<CBInner>,
        config: CircuitBreakerConfig,
    }

    struct CBInner {
        state: CBState,
        consecutive_failures: u32,
        consecutive_successes: u32,
        half_open_permits: u32,  // 当前 HalfOpen 状态下放行的请求数
    }
    ```
    > **设计决策**：state + counters 全部放在同一个 Mutex 里。断路器不是热路径（每次请求仅在开始和结束时各碰一次），Mutex 的开销远小于 Atomic + Mutex 混用带来的 TOCTOU 风险和认知负担。避免并发 `record_success`/`record_failure` 时 counter 和 state 不一致的边界情况。
  - 核心方法：
    - `fn can_execute(&self) -> Result<CBPermit, RuneError>` — 持锁检查：Closed 直接放行；Open 检查是否到了 `reset_timeout_ms`，是则转 HalfOpen 并递增 `half_open_permits`（不超过 `half_open_max_permits`），否则返回 `RuneError::Unavailable`；HalfOpen 检查 permits 余量
    - `fn record_success(&self)` — 持锁：Closed 时重置 failure 计数；HalfOpen 时累加 success 计数，达到 `success_threshold` 后转 Closed 并重置所有计数器
    - `fn record_failure(&self)` — 持锁：Closed 时累加 failure 计数，达到 `failure_threshold` 后转 Open（记录 `Instant::now()`）；HalfOpen 时直接转回 Open
    - `fn state(&self) -> CBState` — 返回当前状态（供监控/日志用）
  - `CBPermit` 是一个 RAII guard，持有时代表有权执行请求，drop 时持锁递减 `half_open_permits`
  - `CircuitBreakerRegistry` 结构体：
    ```rust
    pub struct CircuitBreakerRegistry {
        breakers: DashMap<String, Arc<CircuitBreaker>>,
        config: CircuitBreakerConfig,
    }
    ```
    - `fn get_or_create(&self, caster_id: &str) -> Arc<CircuitBreaker>` — 懒初始化
    - `fn remove(&self, caster_id: &str)` — Caster 断开时清理
    - `fn states(&self) -> Vec<(String, CBState)>` — 列出所有断路器状态（供管理 API 用）
- **依赖**: Task A.1（使用 `CircuitBreakerConfig`）

###### Task A.3: 实现 `RetryInvoker` 包装器

- **新建文件**: `runtime/rune-core/src/retry.rs`
- **改动文件**: `runtime/rune-core/src/lib.rs`（添加 `pub mod retry;`）
- **改动内容**:
  - `RetryInvoker` 结构体：
    ```rust
    pub struct RetryInvoker {
        inner: Arc<dyn RuneInvoker>,
        caster_id: String,
        config: RetryConfig,
        circuit_breaker: Arc<CircuitBreaker>,
    }
    ```
  - `impl RuneInvoker for RetryInvoker`：
    - `invoke_once`：
      1. 检查断路器 `can_execute()`，失败快速返回
      2. 调用 `inner.invoke_once(ctx.clone(), input.clone())`
      3. 成功：`circuit_breaker.record_success()`，返回结果
      4. 失败：若非客户端错误（InvalidInput/NotFound/Cancelled），调用 `circuit_breaker.record_failure()`；判断错误是否 retryable（Timeout 硬编码排除，其余对照 `config.retryable_errors`）
      5. 如果 retryable 且还有重试次数：检查 deadline 剩余预算，计算退避时间 `min(base_delay_ms * multiplier^attempt, max_delay_ms)` 取 full jitter `[0, cap]`，cap 到剩余预算后 `tokio::time::sleep`，醒来后重算 `ctx.timeout` 为实际剩余预算，然后回到步骤 1
      6. 如果不 retryable 或次数耗尽或 deadline 已过：返回最后一个错误
      7. 每次重试生成新的 `request_id`（追加 `-retry-{n}` 后缀），避免与原请求 ID 冲突
      8. 通过 `tracing::warn!` 记录每次重试，包含 `request_id`, `attempt`, `error`, `delay_ms`
    - `invoke_stream`：**不重试**，直接委托 `inner.invoke_stream()`，但仍然检查断路器并记录结果
  - `RuneError` 到 retryable 的映射方法（使用启动时解析的 `HashSet<RuneErrorKind>`，O(1) 查表）：
    ```rust
    fn is_retryable(err: &RuneError, retryable: &HashSet<RuneErrorKind>) -> bool {
        let kind = match err {
            RuneError::Unavailable => RuneErrorKind::Unavailable,
            RuneError::Timeout => RuneErrorKind::Timeout,
            RuneError::Internal(_) => RuneErrorKind::Internal,
            RuneError::ExecutionFailed { .. } => RuneErrorKind::ExecutionFailed,
            RuneError::InvalidInput(_) => RuneErrorKind::InvalidInput,
            RuneError::NotFound(_) => RuneErrorKind::NotFound,
            RuneError::Cancelled => RuneErrorKind::Cancelled,
        };
        retryable.contains(&kind)
    }
    ```
  - **关键设计点**：`RuneContext` 需要 `Clone`（已有 `#[derive(Clone)]`）；`Bytes` 是 `Clone`（引用计数，零拷贝）

### 模块 B：请求追踪 Correlation ID (Request Tracing & Trace Propagation)

#### 设计概述

当前系统存在三个追踪断裂点：
1. **Gate handler**（`runtime/rune-gate/src/handlers/rune.rs` 第 213-218 行）构造 `RuneContext` 时 `context: Default::default()`，没有从 HTTP 请求头提取 trace context
2. **FlowEngine**（`runtime/rune-flow/src/engine.rs` 第 245-250 行）为每个 step 生成独立的 `request_id`（使用 `uuid_simple()` 格式为 `req-{ts_hex}-{seq_hex}`），与 Gate 的 `unique_request_id()` 格式 `r-{ts_hex}-{seq_hex}` 不统一，且没有 parent trace 传递
3. **Proto 的 `context` map** 已经在 `ExecuteRequest` 中存在，但从未被填充 trace 信息

设计方案采用**约定优于配置**的思路：在 `RuneContext.context` HashMap 中使用保留 key 来传递 trace 信息，不改变 `RuneContext` 结构体字段。这样做的好处是：
1. Proto `context: map<string, string>` 已经存在，无需改协议
2. 向后兼容——旧 Caster 收到带 trace 的 context 只是多了几个 key/value，不会 break
3. SDK 侧可以自由选择是否解读这些 key

约定的保留 key：
- `trace_id`：全链路追踪 ID，从 HTTP 请求头提取或自动生成
- `parent_request_id`：Flow 场景下父级 request_id，用于构建调用树
- `span_id`：当前 span 的唯一标识
- `trace_flags`：采样标志（W3C Trace Context 的 trace-flags）

Request ID 格式统一为 `rq-{ts_hex}-{seq_hex}`（Gate 和 Flow 共用同一个生成函数），前缀用 `rq-` 取代当前的 `r-` 和 `req-`。

    ```rust
    /// Generate a unique request ID with the format `rq-{ts_hex}-{seq_hex}`.
    ///
    /// Uses a monotonic counter to ensure uniqueness even within the same
    /// millisecond. Safe for concurrent use across threads.
    pub fn unique_request_id() -> String {
        use std::sync::atomic::{AtomicU64, Ordering};
        static COUNTER: AtomicU64 = AtomicU64::new(0);
        let seq = COUNTER.fetch_add(1, Ordering::Relaxed);
        let ts = now_ms();
        format!("rq-{:x}-{:x}", ts, seq)
    }

    /// Generate a trace ID (128-bit hex, W3C Trace Context compatible).
    pub fn generate_trace_id() -> String {
        use std::sync::atomic::{AtomicU64, Ordering};
        static COUNTER: AtomicU64 = AtomicU64::new(0);
        let ts = now_ms();
        let seq = COUNTER.fetch_add(1, Ordering::Relaxed);
        format!("{:016x}{:016x}", ts, seq)
    }

    /// Generate a span ID (64-bit hex, W3C Trace Context compatible).
    pub fn generate_span_id() -> String {
        use std::sync::atomic::{AtomicU64, Ordering};
        static COUNTER: AtomicU64 = AtomicU64::new(0);
        let seq = COUNTER.fetch_add(1, Ordering::Relaxed);
        let ts = now_ms();
        // Mix ts and seq to get a good 64-bit value
        format!("{:016x}", ts ^ (seq << 32))
    }
    ```

### 模块 C：Store 并发性能优化 (Connection Pool + Key Cache)

#### 设计概述

当前 `RuneStore` 使用单一 `Arc<Mutex<Connection>>` 作为所有数据库操作的入口。在 WAL 模式下，SQLite 本身支持「一写多读」并发，但单 Mutex 使所有操作（包括高频只读的 `verify_key`）退化为全局串行。当多个 HTTP 请求同时命中 Gate 认证中间件时，`verify_key` 排队等锁成为吞吐量瓶颈。

优化策略分三层：**连接池读写分离**、**热路径内存缓存**、**安全性修正**。连接池采用「一写连接 + N 读连接」架构，直接基于 `rusqlite::Connection` 手写池（避免引入 `r2d2` 对 `Send` 约束的适配成本，同时 SQLite 本身不适合通用连接池的多 writer 模型）。热路径上用 `DashMap` + TTL 做一级缓存，`verify_key` 在缓存命中时完全不碰数据库。安全性方面，将 `rand::rng()` 替换为显式 `OsRng` 以满足 CSPRNG 要求，并修复 `output_size` 写死 0 的问题。

迁移路径设计为 `RuneStore` 内部结构变化，但对外 public API（`open()`, `open_in_memory()` 以及所有 `async fn`）签名不变，仅新增可选的配置参数。所有现有测试和调用方无需修改。

- **改动内容**:

  在 `pool.rs` 中定义读写分离的连接池结构：

  ```rust
  pub(crate) struct ConnectionPool {
      writer: Mutex<Connection>,
      readers: Vec<Mutex<Connection>>,
      next_reader: AtomicUsize,
  }
  ```

  - `writer`: 单一写连接，持有 `Mutex`，仅写操作（INSERT/UPDATE/DELETE）使用
  - `readers`: N 个只读连接，每个连接设置 `PRAGMA query_only=ON`，以 round-robin 方式分配

  ```rust
      pub fn new_in_memory(reader_count: usize) -> StoreResult<Self>;
      pub fn writer(&self) -> MutexGuard<'_, Connection>;  // 阻塞获取写锁
      pub fn reader(&self) -> MutexGuard<'_, Connection>;  // round-robin 获取读锁
  }
  ```

  ```rust
  pub async fn verify_key(&self, raw_key: &str, expected_type: KeyType) -> StoreResult<Option<ApiKey>> {
      // 1. 计算 hash
      let key_hash = hash_key(raw_key);

      // 2. 查缓存
      match self.key_cache.get(&key_hash, expected_type) {
          CacheResult::Hit(key) => return Ok(Some(key)),
          CacheResult::NegativeHit => return Ok(None),
          CacheResult::Miss => {}
      }

      // 3. 缓存未命中，查数据库（使用 reader）
      let pool = self.pool.clone();
      let result = tokio::task::spawn_blocking(move || { /* ... SELECT ... */ }).await??;

      // 4. 写回缓存
      match &result {
          Some(key) => self.key_cache.insert(key_hash, key.clone()),
          None => self.key_cache.insert_negative(key_hash),
      }

      Ok(result)
  }
  ```

### 模块 D：Flow 持久化

#### 设计概述

当前 `FlowEngine` 把所有 `FlowDefinition` 存在内存 `HashMap<String, FlowDefinition>` 中。Runtime 重启后所有通过 API 注册的 flow 定义全部丢失，只有硬编码在 `main.rs` 里的 demo flow 能恢复。这对生产环境不可接受。

持久化策略是在 `rune-store` 中新增 `flows` 表，存储 flow 定义的 JSON 序列化。`FlowEngine` 在构造时接受一个 `Arc<RuneStore>` 引用，启动时从数据库加载所有 flow 定义到内存。后续的 `register()` / `remove()` 操作在修改内存的同时同步写入数据库。这种「Write-through + Memory-primary」模式保持了 FlowEngine 现有的同步读取性能（`get()`, `list()` 不走 DB），同时保证持久性。

Flow 执行记录不需要独立持久化——现有的 `tasks` 表已经覆盖了 async flow 执行（task_id 格式为 `flow:{name}`），sync flow 执行的日志可以通过 `call_logs` 表追踪。

  - `created_at`: 首次注册时间

## P1 — 尽快

### 模块 E：Admin Key Bootstrap

#### 设计概述

当前 Rune 的 key 管理存在鸡生蛋问题：非 dev 模式下，创建任何 key 需要通过 `POST /api/v1/keys` API 并提供一个 admin key 进行认证。但第一个 admin key 没有创建方式——启动时数据库里没有任何 key，用户无法调 API 创建 admin key，导致非 dev 模式的 Runtime 实际上无法使用。

Bootstrap 方案提供两个互补的路径：

1. **CLI 命令 `rune key bootstrap`**：直接操作本地 SQLite 数据库文件，绕过 HTTP API 创建首个 admin key。这是推荐的��产部署方式，因��操作发生在本地机器上，不经过���络。

2. **环境变量 `RUNE_AUTH__INITIAL_ADMIN_KEY`**：通过环境变量传入预设的 admin key raw value。Runtime 启动时检测到该变量且数据库中无 admin key 时，自动创建。适用于 Docker/K8s 等无法交互式运行 CLI 的自动化部署场景。

> **安全决策**：不提供 TOML 配置文件中的 `initial_admin_key` 选项。明文密钥写入配置文件极易被误 commit 到 git repo 或泄露到 Docker image layer 中。仅保留环境变量方式——Kubernetes 用户可通过 Secret + envFrom 注入，Docker 用户可通过 `--env-file` 注入。

两种路径都遵循「仅在无 admin key 时生效」的幂等原则，防止误操作覆盖已有的管���权限。

  ```rust
  pub async fn bootstrap(db_path: &str, label: &str, force: bool, json_mode: bool) -> Result<()> {

      let store = RuneStore::open(db_path)
          .with_context(|| format!("failed to open database at '{}'", db_path))?;

          eprintln!();

          eprintln!("  export RUNE_KEY={}", result.raw_key);
  ```

### 模块 F：细粒度 Rate Limiting

#### 设计概述

当前 Rate Limiting 架构是**全局 per-key 单层限流**：所有 Rune 共享同一个 `RateLimitState`（默认 600 req/min），以 `DefaultHasher` 对 Bearer token 取哈希作为 key。这带来三个问题：(1) 高频 Rune 和低频 Rune 无法差异化限流；(2) Caster 端的队列管理全靠 `max_concurrent` semaphore，但默认值为 `usize::MAX >> 1`（约 4.6 * 10^18），等于不限制；(3) `DefaultHasher` 基于 SipHash，Rust 标准库不保证跨版本哈希值一致，如果 Runtime 升级后 rate key 变化，会导致限流窗口异常重置。

新架构采用**三层限流**：第一层 Global per-key（保留现有行为），第二层 Per-rune（根据 rune_name 配置独立的 RPM），第三层 Per-caster（semaphore 合理上限 + 队列溢出拒绝）。三层按顺序依次检查，任一层拒绝即返回 429。Per-rune 限流将 rate key 编码为 `{key_hash}:{rune_name}`，复用现有的滑动窗口算法。哈希算法从 `DefaultHasher` 切换为 `sha2::Sha256` 的前 16 字节（128-bit），截断后格式化为 hex 字符串，确保跨版本、跨平台确定性。

Per-rune 限流规则通过 `rune.toml` 中新增的 `[rate_limit.per_rune]` table 配置，key 为 rune name pattern（支持精确匹配和 `*` 通配），value 为该 Rune 的 RPM。Caster semaphore 默认上限改为 `1024`（与常见 web server worker 数量对齐），并可通过 `CasterAttach.max_concurrent` 显式声明。

### 模块 G：Caster 健康检查 (Health Probe)

#### 设计概述

当前心跳机制（`Heartbeat { timestamp_ms }`）仅检测 gRPC 双向流是否存活——如果 Caster 进程在但处理逻辑完全卡死（比如 OOM 临界、依赖服务挂掉），Runtime 不会感知，照样把请求路由过去。Resolver 选择 Caster 时只看 semaphore permits（`LeastLoadResolver`）或 round-robin，不考虑 Caster 的实际健康状况。

新设计在 Wire Protocol 中新增 `HealthReport` 消息（Caster → Runtime），Caster 可以在每次心跳时附带上报健康数据：`status`（HEALTHY / DEGRADED / UNHEALTHY）、`active_requests`（当前并发数）、`error_rate`（最近窗口的错误率 0.0-1.0）、`custom_info`（自定义 JSON 字符串，如 GPU 利用率、队列深度等）。Runtime 侧 `CasterState` 新增 `HealthInfo` 字段存储最新健康数据。Resolver 通过 `SessionManager` 查询健康状态，`LeastLoadResolver` 过滤掉 UNHEALTHY 的 Caster，`PriorityResolver` 在同优先级中优先选 HEALTHY 而非 DEGRADED。

协议版本协商通过 `AttachAck.protocol_version` 字段实现。Runtime 在 `AttachAck` 中声明自身协议版本（如 `"1.2"`），Caster 收到后检查是否兼容。如果 Caster 发送了 Runtime 不认识的消息类型（如旧 Runtime 不支持 `HealthReport`），protobuf 的 `oneof` 会将其忽略（unknown field），不会报错。这是 protobuf 的向前兼容保证。`supported_features` 字段（目前恒为空）改为真正声明 Runtime 支持的功能列表，如 `["health_probe", "per_rune_rate_limit"]`。

### 模块 H：SDK Proto 同步机制

#### 设计概述

当前项目有一个 proto 源头（`proto/rune/wire/v1/rune.proto`），但三个 SDK 各自管理 proto 文件：Rust SDK 通过 `rune-proto` crate 的 `build.rs` 直接引用 `../../proto/` 目录，天然保持同步；TypeScript SDK 在 `sdks/typescript/proto/rune.proto` 手动拷贝了一份；Python SDK 在 `sdks/python/src/rune/_proto/` 下存放编译后的 `_pb2.py` 和 `_pb2_grpc.py` 文件。这意味着 proto 变更时，TypeScript 拷贝和 Python 生成文件必须手动更新，极易遗忘。

同步策略设计遵循「源头唯一、自动检测」原则。Rust SDK 已经天然同步，不需要改动。TypeScript SDK 保留拷贝文件（`@grpc/proto-loader` 需要本地文件），但由 CI 自动检查拷贝是否与源头一致。Python SDK 同样保留生成文件（避免用户安装时需要 protoc），CI 负责检查生成文件是否最新。不使用 symlink（跨平台兼容性差）或 git submodule（增加仓库管理复杂度），而是用一个简单的 CI job：`diff` 检查 + `protoc` 重新生成后 `diff` 检查。

> **protoc 版本固定**：不同版本的 `grpc_tools.protoc` 可能在生成的 `_pb2.py` 中产生不同的注释或 descriptor bytes，导致 CI 的 `git diff` 检查误报。因此 CI 和 `sync-proto.sh` 脚本中 **必须 pin `grpcio-tools` 版本**（如 `grpcio-tools==1.62.3`），确保本地开发和 CI 环境生成一致的输出。

版本号对齐策略：proto 文件内不编码版本号（protobuf 的 `package` 保持 `rune.wire.v1` 不变，大版本变更才改 `v2`）。SDK 发布版本号与 Rune 主版本号保持一致（v1.2.0），通过 CI 检查各 SDK 的版本文件（`package.json`、`pyproject.toml`、`Cargo.toml`）中的版本号是否与 Git tag 对齐。

## P2 — 技术债

### 模块 I：Relay 数据结构优化

#### 设计概述

当前 Relay 的核心结构是 `DashMap<String, Vec<RuneEntry>>`——rune_name 映射到一个 `Vec`，每个 caster 注册同名 rune 时向 Vec 尾部 push。这导致两个问题：

第一，`remove_caster` 必须遍历所有 rune_name 的所有 entry 做 `retain`，复杂度是 O(R * C)（R = rune 种类数，C = 该 rune 的 caster 数）。当 10000 个 Caster 各注册 10 个 rune 时，单次 `remove_caster` 需要扫描 100000 个 entry。将内层从 `Vec<RuneEntry>` 改为 `HashMap<String, RuneEntry>`（caster_id → entry），`remove_caster` 只需为每个 rune_name 做一次 O(1) 的 `remove(caster_id)`，总复杂度降到 O(R)。对于本地注册（`caster_id = None`），使用合成 key（如 `"__local__{uuid}"`）保持接口统一。

第二，`write_lock: Mutex<()>` 与 DashMap 的组合存在一致性窗口：`resolve` 和 `find` 不持 `write_lock`，可能在 `remove_caster` 执行到一半时读到不一致的状态（比如 entries 中已删但 gate_path_index 还在）。鉴于读路径是热路径（每个 HTTP 请求都要走），给读路径加锁会严重影响吞吐。因此选择**接受最终一致**的策略：(a) `resolve` 失败时上层已有 fallback（返回 404/503），(b) 不一致窗口极短（微秒级），(c) 增加一个 `generation: AtomicU64` 计数器，每次写操作自增，读操作可选校验。将 `Mutex<()>` 替换为 `tokio::sync::RwLock<()>` 不带来收益（写操作已串行化，读操作不持锁）——保持现有 `Mutex` 做写串行化即可。

同时引入反向索引 `caster_index: DashMap<String, Vec<String>>`（caster_id → `Vec<rune_name>`），在 `register` 时维护，在 `remove_caster` 时直接查出该 caster 注册了哪些 rune，避免全表扫描。

     ```rust
     pub fn remove_caster(&self, caster_id: &str) {
         let _guard = self.write_lock.lock().unwrap_or_else(|e| e.into_inner());
     
         // 通过反向索引找到该 caster 注册了哪些 rune
         let rune_names = self.caster_index.remove(caster_id)
             .map(|(_, names)| names)
             .unwrap_or_default();
     
         let mut gate_keys_to_check: Vec<(String, String)> = Vec::new();
     
         for rune_name in &rune_names {
             if let Some(mut entry_map) = self.entries.get_mut(rune_name) {
                 // 在 remove 之前收集 gate path 信息
                 if let Some(e) = entry_map.get(caster_id) {
                     if let Some(ref gate) = e.config.gate {
                         gate_keys_to_check.push((
                             format!("{}:{}", gate.method, gate.path),
                             rune_name.clone(),
                         ));
                     }
                 }
                 // O(1) 删除
                 entry_map.remove(caster_id);
             }
         }
     
         // 清理空 entry map
         for rune_name in &rune_names {
             self.entries.remove_if(rune_name, |_, v| v.is_empty());
         }
     
         // 清理 gate_path_index
         for (key, rune_name) in gate_keys_to_check {
             let still_exists = self.entries.get(&rune_name)
                 .map(|em| em.values().any(|e| {
                     e.config.gate.as_ref()
                         .map(|g| format!("{}:{}", g.method, g.path))
                         .as_deref() == Some(&key)
                 }))
                 .unwrap_or(false);
             if !still_exists {
                 self.gate_path_index.remove(&key);
             }
     ```

### 模块 K：Error 类型细化 + Axum 路径语法修复

#### 设计概述

当前 `RuneError` 有两个主要不足：(1) Gate 层返回错误时不携带 `request_id`，用户看到 `{"error": {"code": "TIMEOUT", "message": "timeout"}}` 无法定位具体请求；(2) `Internal(anyhow::Error)` 是个"垃圾桶" variant，丢失了结构化信息。

对于 `request_id` 的携带方式，有两个选择：
- **选项 A**: 在每个 `RuneError` variant 中添加 `request_id` 字段——侵入性太强，`RuneError` 应该是通用错误类型，不应绑定 HTTP 层概念
- **选项 B（推荐）**: 在 Gate 的 error response 层面注入 `request_id`，即改 `error_response` 函数和 `map_error` 函数，接受 `request_id` 参数，在 JSON 响应体中包含它

选择方案 B。`RuneError` 保持通用，Gate 层（HTTP 边界）负责将 `request_id` 附加到错误响应体。

对于 `Internal(anyhow::Error)` 的细分：新增 `RateLimited { retry_after_secs: u64 }` 和 `CircuitOpen { rune_name: String }` 两个 variant（配合模块 A 的 rate limit 和 circuit breaker）。同时新增 `Unauthorized` 和 `Forbidden`——当前 Gate middleware 直接构造 HTTP response，不经过 `RuneError`，但统一 error 类型有利于 Flow 层错误传播。

Axum 路径语法是一个简单的全局搜索替换：`:name` → `{name}`。Axum 0.7 中 `:name` 是 deprecated 但仍可工作，未来版本可能移除。

### 模块 L：Pilot & 自动扩缩容 (Pilot & Auto-Scaling)

#### 设计概述

模块 G 的 `HealthReport` 解决了「Caster 是否健康」的判断（HEALTHY / DEGRADED / UNHEALTHY），但无法回答「Caster 负载有多高」和「是否需要扩缩容」。负载的评估只有 Caster 自身最清楚（CPU / 内存 / 队列深度 / GPU 利用率等），Runtime 从外部推测（数 semaphore permits、算延迟趋势）是不可靠的。

**三层职责分离**：

| 层 | 职责 | 谁做 |
|----|------|------|
| 感知 | 判定自身是否过载、计算 pressure | **Caster**（开发者提供 load_reporter 回调） |
| 评估 + 信号 | 基于 Caster 声明的判断函数评估，产生扩/缩容信号 | **Runtime** |
| 执行 | 接收信号，启停 Caster 进程 | **Pilot**（本地伴生进程） |

**Pilot 解决的核心拓扑问题**：Runtime 可能在云端，Caster 在本地。Runtime 无法直接在 Caster 所在机器上启停进程。Pilot 作为本地代理，通过 Session 协议连接 Runtime 接收信号，在本地执行扩缩容。

**Pilot 是一种特殊的内置 Caster**——复用 Session 协议连接 Runtime，通过 `CasterAttach.role = "pilot"` 区分身份。不注册任何 Rune，只参与管理通道。

#### 双层架构

Pilot 和 Caster 的交互分为两层，职责清晰隔离：

```
┌─────────────────────────────────────────────────┐
│ Rune 协议层（gRPC Session，通过 Runtime）        │
│                                                 │
│  Caster ──HealthReport(pressure)──→ Runtime     │  负载上报
│  Runtime ──ScaleSignal──→ Pilot                 │  扩容指令
│  Runtime ──ShutdownRequest──→ Caster            │  缩容指令
│                                                 │
│  所有业务调用、负载评估、扩缩容信号走此层         │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│ 本地基础设施层（Unix socket，不经 Runtime）       │
│                                                 │
│  Caster ──注册(PID, spawn_cmd)──→ Pilot         │  进程注册
│  Caster ──注销──→ Pilot                         │  优雅退出
│  Pilot ──kill(PID)──→ OS                        │  强制清理
│                                                 │
│  纯 OS 级进程生命周期管理，不涉及业务逻辑        │
└─────────────────────────────────────────────────┘
```

**设计原则**：Rune 的「Caster 之间通过 Runtime 通信」是**业务逻辑层**的约束——调用 Rune、传递数据、负载上报、扩缩容信号都走 Runtime。但进程生命周期管理（PID、spawn_command、SIGTERM）是 OS 级基础设施关注点，天然是本地的，不应依赖远程 Runtime 的可用性。类比 Kubernetes：Pod 之间通过 Service 通信，但 kubelet 在本地管理 Pod 生命周期。

**缩容路径**不经 Pilot：Runtime 直接通过 gRPC Session 发 `ShutdownRequest` 给目标 Caster，Caster 优雅退出。只有 Caster 超时未退出时，Pilot 才作为 fallback 通过本地 PID 强制 kill。

#### Pilot 是什么

**Pilot 是 `rune` CLI 的一个子命令（`rune pilot daemon`），单一 Rust 二进制。** 不是 SDK 各语言各自实现的进程。Python Caster、TypeScript Caster、Rust Caster 发现和连接的都是同一个 Pilot 进程。

**Pilot 职责边界**：
1. 本地注册表 —— 知道本机哪些 Caster 进程、PID、启动命令（通过本地 socket 接收注册）
2. 接收 Runtime ScaleSignal —— 通过 gRPC Session（Rune 协议层）
3. 执行 spawn —— 扩容时启动新 Caster 进程
4. Fallback force-kill —— 缩容超时时通过 PID 强制终止

Pilot **不做**：调度、负载评估、转发业务流量、主动缩容（缩容由 Runtime 直接通知 Caster）。

#### Pilot 幂等发现协议

**核心约束：无论 Python / TypeScript / Rust SDK 还是 CLI，发现和启动 Pilot 的行为必须幂等——所有语言的 Caster 必须发现同一个 Pilot 进程。**

发现协议基于文件系统约定（跨语言通用）：

| 文件 | 用途 |
|------|------|
| `~/.rune/pilot.lock` | Advisory file lock，防止并发 spawn 竞争 |
| `~/.rune/pilot.pid` | Pilot 进程 PID，用于存活检测 |
| `~/.rune/pilot.sock` | Unix domain socket，Pilot 本地通信端点 |

```
Caster 启动
    │
    ▼
有 @caster.scaler？── 没有 ──→ 直连 Runtime（与现有行为一致）
    │ 有
    ▼
┌──────────────────────────────────────────┐
│ 幂等发现流程（所有 SDK/CLI 共用）          │
│                                          │
│ 1. flock(~/.rune/pilot.lock)  ← 加锁    │
│ 2. 读 ~/.rune/pilot.pid                  │
│ 3. PID 对应进程存活？                     │
│    ├── 是 → 释放锁 → 连接 pilot.sock     │
│    └── 否 → spawn `rune pilot daemon`    │
│              等待 pilot.sock 可用         │
│              释放锁                       │
│ 4. 向 Pilot 注册（PID + spawn_command）   │
└──────────────────────────────────────────┘
```

**幂等保证**：
- **file lock** 确保并发启动的多个 Caster 只有一个会 spawn Pilot
- **PID 文件 + 进程存活检测** 确保不会重复启动
- **Unix socket** 作为通信端点，连接即验证——socket 不存在或连接失败 = Pilot 不可用
- Pilot 故障时，下一个尝试注册的 Caster 检测到 socket 断开 → 重新走发现流程 → 自动恢复
- 所有 Caster 退出后，Pilot 检测到注册表为空 → 自动退出 → 清理 PID 文件和 socket
- 没有 `@caster.scaler` 的 Caster 完全不参与此流程，行为与 v1.1.0 一致

- **改动文件**: `runtime/rune-cli/src/commands/pilot.rs`（新文件）
- **改动文件**: `runtime/rune-cli/src/commands/mod.rs`（注册子命令）
- **改动内容**:
  - **`rune pilot daemon`**：Pilot 守护进程，所有语言的 SDK 共用这一个进程
    1. 写入 PID 到 `~/.rune/pilot.pid`
    2. 监听 `~/.rune/pilot.sock`（Unix domain socket），接受 Caster 本地注册
    3. 以 `role: "pilot"` 连接 Runtime（通过 Session 协议），不注册 Rune
    4. 维护本地注册表：`{ caster_id, pid, spawn_command, shutdown_signal }`
    5. 接收 Runtime 的 `ScaleSignal`：
       - `ScaleAction::Up` → 执行 `spawn_command` 启动新 Caster 实例，记录 PID
       - `ScaleAction::Down` with force_kill → 根据 caster_id 查本地注册表找 PID，发 `SIGKILL`（fallback，仅当 Runtime 的 ShutdownRequest 超时未生效时触发）
    - 注意：正常缩容由 Runtime 直接发 `ShutdownRequest` 给 Caster，不经 Pilot。Pilot 只处理强制终止的 fallback 场景
    6. 定时心跳检测已注册 Caster 进程存活（kill(pid, 0)），清理死进程
    7. 注册表为空 → 自动退出 → 清理 PID 文件和 socket
    8. 启动参数：`rune pilot daemon --runtime <addr>` 指定 Runtime 地址
  - **`rune pilot status`**：查看本地 Pilot 状态（PID、注册的 Caster 列表、连接的 Runtime）
  - **`rune pilot stop`**：优雅停止 Pilot
  - 进程生命周期文件管理：
    - 启动时：加 flock → 写 PID → 监听 socket → 释放 flock
    - 退出时（正常/信号）：删除 `pilot.pid` + `pilot.sock`（通过 shutdown hook）
    - 崩溃时：PID 文件残留，下次 Caster 发现协议检测到 PID 不存活 → 清理并重新 spawn

## 模块 I：Relay 数据结构优化

### 设计概述

当前 Relay 的核心结构是 `DashMap<String, Vec<RuneEntry>>`——rune_name 映射到一个 `Vec`，每个 caster 注册同名 rune 时向 Vec 尾部 push。这导致两个问题：

第一，`remove_caster` 必须遍历所有 rune_name 的所有 entry 做 `retain`，复杂度是 O(R * C)（R = rune 种类数，C = 该 rune 的 caster 数）。当 10000 个 Caster 各注册 10 个 rune 时，单次 `remove_caster` 需要扫描 100000 个 entry。将内层从 `Vec<RuneEntry>` 改为 `HashMap<String, RuneEntry>`（caster_id → entry），`remove_caster` 只需为每个 rune_name 做一次 O(1) 的 `remove(caster_id)`，总复杂度降到 O(R)。对于本地注册（`caster_id = None`），使用合成 key（如 `"__local__{uuid}"`）保持接口统一。

第二，`write_lock: Mutex<()>` 与 DashMap 的组合存在一致性窗口：`resolve` 和 `find` 不持 `write_lock`，可能在 `remove_caster` 执行到一半时读到不一致的状态（比如 entries 中已删但 gate_path_index 还在）。鉴于读路径是热路径（每个 HTTP 请求都要走），给读路径加锁会严重影响吞吐。因此选择**接受最终一致**的策略：(a) `resolve` 失败时上层已有 fallback（返回 404/503），(b) 不一致窗口极短（微秒级），(c) 增加一个 `generation: AtomicU64` 计数器，每次写操作自增，读操作可选校验。将 `Mutex<()>` 替换为 `tokio::sync::RwLock<()>` 不带来收益（写操作已串行化，读操作不持锁）——保持现有 `Mutex` 做写串行化即可。

同时引入反向索引 `caster_index: DashMap<String, Vec<String>>`（caster_id → `Vec<rune_name>`），在 `register` 时维护，在 `remove_caster` 时直接查出该 caster 注册了哪些 rune，避免全表扫描。

##### Task I.3: 重写 `remove_caster` 方法

- **改动文件**: `runtime/rune-core/src/relay.rs`
- **改动内容**:
  1. 利用 `caster_index` 直接获取该 caster 注册的所有 rune_name:
     ```rust
     pub fn remove_caster(&self, caster_id: &str) {
         let _guard = self.write_lock.lock().unwrap_or_else(|e| e.into_inner());
     
         // 通过反向索引找到该 caster 注册了哪些 rune
         let rune_names = self.caster_index.remove(caster_id)
             .map(|(_, names)| names)
             .unwrap_or_default();
     
         let mut gate_keys_to_check: Vec<(String, String)> = Vec::new();
     
         for rune_name in &rune_names {
             if let Some(mut entry_map) = self.entries.get_mut(rune_name) {
                 // 在 remove 之前收集 gate path 信息
                 if let Some(e) = entry_map.get(caster_id) {
                     if let Some(ref gate) = e.config.gate {
                         gate_keys_to_check.push((
                             format!("{}:{}", gate.method, gate.path),
                             rune_name.clone(),
                         ));
                     }
                 }
                 // O(1) 删除
                 entry_map.remove(caster_id);
             }
         }
     
         // 清理空 entry map
         for rune_name in &rune_names {
             self.entries.remove_if(rune_name, |_, v| v.is_empty());
         }
     
         // 清理 gate_path_index
         for (key, rune_name) in gate_keys_to_check {
             let still_exists = self.entries.get(&rune_name)
                 .map(|em| em.values().any(|e| {
                     e.config.gate.as_ref()
                         .map(|g| format!("{}:{}", g.method, g.path))
                         .as_deref() == Some(&key)
                 }))
                 .unwrap_or(false);
             if !still_exists {
                 self.gate_path_index.remove(&key);
             }
         }
     
         self.generation.fetch_add(1, Ordering::Release);
     }
     ```
  2. 复杂度从 O(R_total * C_max) 降至 O(R_this_caster)——只遍历当前 caster 注册的 rune，每个 rune 内做 O(1) HashMap remove

## 模块 J：去掉 async_trait，使用原生 async fn in trait

### 设计概述

Rust 1.75 (2023-12-28) 稳定了 `async fn in trait`，当前项目使用 rustc 1.93.0，完全满足要求。`async_trait` 宏的核心代价是每次调用都会产生一个 `Box<dyn Future>` 堆分配——在 `RuneInvoker::invoke_once` 这条热路径上，每个 HTTP 请求至少触发一次。去掉宏后编译器可以进行静态分发（monomorphization），消除热路径上的堆分配。

然而，核心难点在于 **`dyn RuneInvoker` 的兼容性**。Rune 大量使用 `Arc<dyn RuneInvoker>` 作为类型擦除的 invoker handle（`relay.rs` 的 `RuneEntry.invoker`、`rune-gate` handler 的函数参数等）。原生 `async fn in trait` **不支持直接通过 `dyn Trait` 调用**，因为 async fn 返回的 Future 类型是 impl-specific 的，无法做 vtable 分发。

解决方案有三条路径：

1. **`trait_variant::make_variant` 宏**（`trait-variant` crate by dtolnay）——自动生成一个 `dyn`-compatible 的伴生 trait，其 async fn 返回 `Box<dyn Future>`。这样核心 trait 定义使用原生 async，而需要 `dyn` 分发的地方使用伴生 trait。代价是引入一个新 proc-macro 依赖。

2. **手写 blanket impl + 手动返回 `Box<dyn Future>`**——定义一个 `DynRuneInvoker` trait，其方法返回 `Pin<Box<dyn Future>>`，为所有 `T: RuneInvoker` 实现 `DynRuneInvoker`。`RuneEntry` 存 `Arc<dyn DynRuneInvoker>`。这样核心 trait 是零开销的原生 async，`dyn` 路径用手写的 wrapper trait。

3. **保留 `async_trait` 仅用于需要 `dyn` 分发的 trait（`RuneInvoker`），移除其他 trait 上的 `async_trait`**——最小化改动，只在确定不需要 `dyn` 的 trait 上使用原生 async fn。

**选择方案 2**。理由：(a) 不引入新 proc-macro 依赖，(b) `dyn` 分发路径本身就有 vtable 间接调用开销，再加一层 `Box<dyn Future>` 是可接受的，(c) 但对 `LocalInvoker` 的直接（静态分发）调用可以完全消除堆分配——然而当前架构中所有调用都走 `Arc<dyn RuneInvoker>` 动态分发，因此实际收益有限。

**重新评估**：鉴于 Rune 架构中 `RuneInvoker` 始终通过 `Arc<dyn RuneInvoker>` 使用，无论用哪种方案，`dyn` 分发路径上的 `Box<dyn Future>` 都不可避免。因此实际收益集中在：
- `StreamRuneHandler` trait——仅在 `LocalStreamInvoker` 内部通过 `Arc<dyn StreamRuneHandler>` 使用，同样是 dyn 路径
- `KeyVerifier` trait——通过 `Arc<dyn KeyVerifier>` 使用

**最终决策**：采用**方案 3（渐进式）**——对于不通过 `dyn` 使用的 trait 去掉 `async_trait`（如测试中的 mock 实现），对于核心需要 `dyn` 的 trait 暂保留 `async_trait`，同时准备好在 Rust 稳定 `async fn in dyn Trait`（RFC 3745，预计 2025-2026 稳定）后的迁移路径。在此基础上，对可以改为静态分发的调用点进行泛型化。

**补充**：Rust 1.75 虽然稳定了 async fn in trait，但仅限于**非 dyn-safe** 场景。当前 crate 中以下 trait 需要 dyn 使用，因此**暂时无法直接去掉 async_trait**：
- `RuneInvoker` → `Arc<dyn RuneInvoker>`
- `StreamRuneHandler` → `Arc<dyn StreamRuneHandler>`
- `KeyVerifier` → `Arc<dyn KeyVerifier>`

为了在当前 Rust 版本上获得收益，采用**双 trait 模式（方案 2 的变体）**：

```rust
// 原生 async trait（零 heap alloc，但不 dyn-safe）
pub trait RuneInvoker: Send + Sync {
    fn invoke_once(&self, ctx: RuneContext, input: Bytes)
        -> impl Future<Output = Result<Bytes, RuneError>> + Send;
    fn invoke_stream(&self, ctx: RuneContext, input: Bytes)
        -> impl Future<Output = Result<mpsc::Receiver<Result<Bytes, RuneError>>, RuneError>> + Send;
}

// dyn-safe wrapper trait（供 Arc<dyn DynRuneInvoker> 使用）
pub trait DynRuneInvoker: Send + Sync {
    fn invoke_once(&self, ctx: RuneContext, input: Bytes)
        -> Pin<Box<dyn Future<Output = Result<Bytes, RuneError>> + Send + '_>>;
    fn invoke_stream(&self, ctx: RuneContext, input: Bytes)
        -> Pin<Box<dyn Future<Output = Result<mpsc::Receiver<Result<Bytes, RuneError>>, RuneError>> + Send + '_>>;
}

// Blanket impl: 任何实现了 RuneInvoker 的类型自动实现 DynRuneInvoker
impl<T: RuneInvoker> DynRuneInvoker for T {
    fn invoke_once(&self, ctx: RuneContext, input: Bytes)
        -> Pin<Box<dyn Future<Output = Result<Bytes, RuneError>> + Send + '_>> {
        Box::pin(RuneInvoker::invoke_once(self, ctx, input))
    }
    fn invoke_stream(&self, ctx: RuneContext, input: Bytes)
        -> Pin<Box<dyn Future<Output = Result<mpsc::Receiver<Result<Bytes, RuneError>>, RuneError>> + Send + '_>> {
        Box::pin(RuneInvoker::invoke_stream(self, ctx, input))
    }
}
```

这样 `LocalInvoker`, `LocalStreamInvoker`, `RemoteInvoker` 只需实现 `RuneInvoker`（原生 async，无宏），而 `RuneEntry` 中存 `Arc<dyn DynRuneInvoker>`。性能上与 `async_trait` 等价（dyn 路径仍有 Box），但代码更干净、编译时无宏展开开销、IDE 支持更好、future 类型在栈上。

## 模块 K：Error 类型细化 + Axum 路径语法修复

### 设计概述

当前 `RuneError` 有两个主要不足：(1) Gate 层返回错误时不携带 `request_id`，用户看到 `{"error": {"code": "TIMEOUT", "message": "timeout"}}` 无法定位具体请求；(2) `Internal(anyhow::Error)` 是个"垃圾桶" variant，丢失了结构化信息。

对于 `request_id` 的携带方式，有两个选择：
- **选项 A**: 在每个 `RuneError` variant 中添加 `request_id` 字段——侵入性太强，`RuneError` 应该是通用错误类型，不应绑定 HTTP 层概念
- **选项 B（推荐）**: 在 Gate 的 error response 层面注入 `request_id`，即改 `error_response` 函数和 `map_error` 函数，接受 `request_id` 参数，在 JSON 响应体中包含它

选择方案 B。`RuneError` 保持通用，Gate 层（HTTP 边界）负责将 `request_id` 附加到错误响应体。

对于 `Internal(anyhow::Error)` 的细分：新增 `RateLimited { retry_after_secs: u64 }` 和 `CircuitOpen { rune_name: String }` 两个 variant（配合模块 A 的 rate limit 和 circuit breaker）。同时新增 `Unauthorized` 和 `Forbidden`——当前 Gate middleware 直接构造 HTTP response，不经过 `RuneError`，但统一 error 类型有利于 Flow 层错误传播。

Axum 路径语法是一个简单的全局搜索替换：`:name` → `{name}`。Axum 0.7 中 `:name` 是 deprecated 但仍可工作，未来版本可能移除。

## 跨模块依赖关系

```
模块 I (Relay 数据结构)  ←独立→  模块 K (Error + Axum)
        ↓ (relay.rs 中 RuneEntry.invoker 类型变了)
模块 J (async_trait 迁移) ←依赖→ 模块 I (I.5 的 Resolver 参数 &[RuneEntry])
```

**推荐实施顺序**: K → I → J

- **K 最独立**：Error variant 新增和 Axum 语法替换不影响 I 和 J
- **I 次之**：Relay 数据结构重构是 J 的前提（因为 J.3 需要修改 `RuneEntry.invoker` 类型，如果 I 先做完，J 只需在新结构上改；反之两个 PR 会冲突）
- **J 最后**：async_trait 迁移涉及面最广，且需要所有使用 `Arc<dyn RuneInvoker>` 的地方同步改动，适合在 I 和 K 稳定后单独做

每个模块可以独立一个 feature branch 和 PR。
